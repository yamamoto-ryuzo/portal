# 第2章 ソースをSSOT（Single Source of Truth）とする設計

## 要旨（先に結論）
この章は、RAG（Retrieval‑Augmented Generation）運用において「SSOT（Single Source of Truth）」を中心に据え、モデルのコンテキスト制約とオーケストレーション設計を一致させることで、コスト・レイテンシ・検証負荷を抑えつつ高精度な生成を実現するための実務方針を示します。

## 本章で使う基本用語（初心者向けガイド）
専門用語が多いため、まずは以下の基本概念を押さえてください。

**システム全般・基本概念**
- **LLM（大規模言語モデル）**: ChatGPT（OpenAI）や Gemini（Google）、Claude（Anthropic）などのAIそのものを指します。
- **RAG（検索拡張生成）**: AI（LLM）が知らない最新情報や社内データを、検索システムを使って探し出し、その回答の「カンペ（情報源）」としてAIに渡して答えさせる仕組みです。
- **RAGソース**: RAGが検索し、AIに渡すためのデータの元となる**「文書ファイル全体」**のことです（例：100ページの社内マニュアルPDF、議事録のWordファイルなど）。
- **SSOT（Single Source of Truth / 唯一の正しい情報源）**: 「これが絶対に正しい」と組織内で合意されたマスターデータのこと。RAGにゴミデータ（古い情報や間違った情報）を渡すとAIも間違えるため、「AIにはSSOTだけを渡す」ことが極めて重要になります。

**データの分割と要約に関する用語**
- **チャンク**: 長い文章（RAGソース）をAIが読みやすいように、数百文字〜数千文字程度の「意味のかたまり」に**切り分けたブロック**のことです。
- **生チャンク（詳細参照）**: RAGソースを切り分けただけで、**中身を要約せず原文そのままのブロック**のことです。詳細な情報が残っていますが、文字数を多く消費します。
- **凝縮要約（インデックス）**: 原文（生チャンク）の内容をAIが短く100〜300文字程度に**要約したブロック**のことです。検索を速く・安く行うための「見出し」や「目次」として使います。

**AIの処理量とプロンプト構成に関する用語**
- **トークン（Token）**: AIがテキストを読み書きする際の「文字の数え方の単位」です。英語なら `1トークン ≒ 約4文字`、日本語なら `1トークン ≒ 約1〜2文字` の目安です。
- **コンテキスト（コンテキストウィンドウ）**: AIが「1回の会話で一度に読める・記憶できる文章量の限界」のことです。例えば「コンテキスト200K（20万トークン）」のAIなら、本数冊分（約80万文字）の資料を一度に読んで回答できます。
- **プロンプトキャッシュ（Prompt Caching）**: 毎回同じ長い資料（SSOT）をAIに読ませるとお金と時間がかかりますが、一度読んだ資料をAI側に「一時記憶（キャッシュ）」させておくことで、2回目以降の回答を安く・爆速にする最新技術です。
- **システム指示（システムプロンプト）**: 「あなたは優秀なアシスタントです」「必ず敬語で答えてください」といった、AIの振る舞いやルールを規定する大前提の指示文のことです。
- **チャット履歴**: ユーザーとAIの過去の会話のやりとりのことです。文脈を保つために、直近の会話をAIに毎回一緒に渡す必要があります。
- **出力予備（モデルへ確保するバッファ）**: AIが回答（出力）するための空きスペースです。AIのコンテキスト上限ギリギリまで資料（入力）を詰め込むと、AIが回答を書くための余白がなくなりエラーになるため、意図的に空けておく余裕（バッファ）のことです。

**（参考）一連の処理フロー（言葉がどう使われるか）**
実際のRAGシステムでは、上記の言葉が以下の順番で組み合わさって動きます。
1. **ユーザーの入力**: ユーザーがチャット画面で質問（クエリ）を入力する。
2. **AIのRAG検索（インデックス検索）**: ユーザーの質問に関連しそうなデータを「凝縮要約」の中から高速に探し出す。
3. **AIのRAG抽出（生チャンク取得）**: 検索でヒットした要約の元となっている詳しい原文（生チャンク）をデータベースから取り出す。
4. **元のRAGソースの自動取得（オートローディング）**: 
   - 抽出した生チャンクに記録されている「ファイルの保存場所」の情報を使い、システムが裏側で**「元のRAGソース（画像や図表を含むPDFなどのファイル全体）」**を自動的に引っ張ってきます。
   - （※画像を含まない単純なテキスト質問の場合は、生チャンクだけをAIに渡すことで処理を軽くすることもあります）
5. **プロンプト合成**: AIに渡すための最終的な「カンペ」を作る。[システム指示] ＋ [引っ張ってきたRAGソース全体（SSOT）] ＋ [チャット履歴] ＋ [ユーザーの入力] を、コンテキスト上限を超えないように1つの文章に合体させる。
6. **AIの回答**: 合体されたカンペを読んだLLMが、出力予備（バッファ）を使って最終的な答えを生成し、画面に返す。

## 背景と目的
・RAGソースは量だけでなく「整備されたSSOT」を起点とすることが品質と検証性を担保する最短ルートです。  
・モデルのコンテキスト上限（SSOT参照）を踏まえ、オーケストレーション側でチャンク化・再ランク・段階注入を行うことが本設計の核です。

## コンテキスト予算の判定式（設計式）
設計上の合算式（トークン換算）:
$$total\_context\_tokens \approx orchestration\_tokens + RAG\_references\_tokens + file\_tokens + chat\_history\_tokens + prompt\_instructions\_tokens + output\_reserved\_tokens$$

実務試算例（標準モデル想定 — 入力200K / 出力200K）: 合計コンテキスト 400K トークンのうち、プロンプト予算は安全マージンを考慮して50〜70%を推奨（= 200K〜280K トークン）。
※文字数換算の目安：英語主体なら 1 token $\approx$ 約4文字、日本語主体なら 1 token $\approx$ 約1〜2文字。以下は「1 token $\approx$ 4文字換算」での最大目安字数を併記します。

例（概念）: オーケストレーション 500 tokens (約2,000字) + RAG参照 10,000 tokens (約40,000字) + ファイル抜粋 2,000 tokens (約8,000字) + チャット履歴 1,000 tokens (約4,000字) + 指示 400 tokens (約1,600字) + 出力予備 5,000 tokens (約20,000字) = 合計 18,900 tokens（約75,600文字、標準予算の約9.5%）。

設計方針: 使用モデルのSSOTに基づき、合計がモデルコンテキストの50〜70%以内に収まるよう予算配分する。ここでは標準想定として「入力200K / 出力200K」を前提に設計例を示す。

**標準モデル（入力200K / 出力200K 想定） — 推奨プロンプト配分（例）**:  
- システム指示: 1,000〜5,000 tokens（約4,000〜20,000字）  
- 凝縮要約（インデックス）: 合計 10,000〜30,000 tokens（約4万〜12万字。例: 20件×500 tokens）  
- 生チャンク合計（詳細参照）: 20,000〜60,000 tokens（約8万〜24万字）  
- チャット履歴: 1,000〜10,000 tokens（約4,000〜40,000字）  
- 出力予備（モデルへ確保するバッファ）: 5,000〜20,000 tokens（約2万〜8万字）  
上記合計が 200K〜280K tokens（約80万〜112万文字）の範囲に収まるよう調整してください。

## 影響と設計パターン（大容量コンテキスト前提のSSOT中心設計）
標準（200K）および高性能（500K）という広大なコンテキストウィンドウを前提とする場合、RAGの設計パターンは従来の「極限まで削る」アプローチから「確実なSSOTを広く取り込む」アプローチへシフトします。

1. **Massive Context RAG（広域一括注入）**: 凝縮要約だけでなく、関連する可能性のある生チャンクを数十〜数百件規模で一括してプロンプトに注入します。検索漏れ（False Negative）のリスクを下げ、LLM自身の文脈理解力にフィルタリングを委ねます。
2. **Context rolling（段階注入）**: 500Kを超える極端な長文（例：巨大なコードベース全体や全履歴）の処理には、依然としてロールングで段階的に情報を注入する設計が必要です。
3. **Few‑shot とシステム固定文**: スタイルや出力フォーマットの指定、大量のFew-shot例をシステム指示に含めてもコンテキストを圧迫しません。頻出挙動はプロンプトで十分に制御可能であり、軽量FT（LoRA等）の必要性は低下します。
4. **キャッシュと差分インジェスト**: 広大なコンテキストを毎回フルロードするとコストと遅延が増大します。承認済みSSOTはプロンプトキャッシュ（Prompt Caching）を活用し、更新は差分インジェストで同期します。
5. **コスト監視とバジェット**: 巨大コンテキストモデルは入力トークン単価の管理が重要です。プロンプトキャッシュのヒット率を最大化するよう、静的（不変）なSSOTブロックと動的（クエリ依存）なブロックを分離するプロンプト構成が必須です。

### （重要）テキストRAGの限界と「画像・図表の欠落」問題
M365のMicrosoft Graphや一般的なベクトルDBを用いたRAG検索は、**純粋な「テキストデータ（文字情報）」のみ**を抽出します。
そのため、LLM（回答AI）自身が GPT-4o や Gemini などのように「画像を理解できる能力（マルチモーダル）」を持っていたとしても、**AIに渡されるカンペ（生チャンク）の中にそもそも画像が存在しないため、画像やグラフに基づいた回答は不可能**になります。

**解決策のアプローチ：**
- **テキストRAG（通常）**: 文字ベースの検索に割り切る。画像内の情報は事前OCRや図表要約AIを使って「文字として」チャンクに埋め込んでおく。
- **フルソース読込（マルチモーダル対応）**: もし画像やレイアウトの理解が必須なクエリの場合は、RAGによる「生チャンク」の抽出で済ませるのではなく、**ヒットしたRAGソース（元のPDFやWordファイル全体）を丸ごとAIのコンテキストに投げ込む（再読込する）**必要があります。
  - *自動化の理想*: システム側で「検索でヒットした生チャンクのメタデータ（ファイルパス等）」を読み取り、**裏側で自動的に元のファイル全体をフェッチしてLLMに渡す自動パイプライン**を独自構築することが可能です。大容量コンテキスト（200K〜500K）の時代だからこそ実現できる強力なアーキテクチャです。
  - *【M365 Copilotの現実と構造的限界】*: 
    現在の標準的な M365 Copilot（Graph Grounding）は、セキュリティと速度を優先したSaaS製品であり、以下の構造的な制約（ブラックボックス）を抱えています。
    1. **Graph APIが返すのは「テキストのみ」**: Microsoft GraphがSharePoint等から検索してLLMに渡すのは「テキスト化された生チャンクの文字列」と「ファイルのURL」だけです。画像やレイアウトのバイナリデータは渡されません。
    2. **LLMはファイルに直接アクセスできない**: 「ファイルのURL」をLLMに渡しても、LLM自身はWebブラウザのように社内のURLにアクセスしてPDFをダウンロードする機能や権限を持っていません。
    3. **オートローディング（自動再読込）のパイプラインがない**: M365 Copilotの内部には「URLを受け取って、裏でこっそり元ファイルをダウンロードし、画像に変換してLLMに投げ直す」という仕組みが実装されていません。ユーザーや管理者がこの内部仕様を改造することも不可能です。
    
    **結論として、M365 Copilotを使う限り、裏で動くモデルがどれほど優秀なマルチモーダルAI（GPT-4o等）になっても、RAGとして画像を理解して回答することはできません。**
    これを解決するには、「ユーザーが手動でチャット画面に該当ファイルを直接添付（参照指定）する」か、「事前にWordやPowerPoint側で図表のAltテキスト（代替テキスト）を詳細に書き込んでおく（テキストRAGへの歩み寄り）」というアナログな運用カバーが必須となります。
    
    **※補足：M365 Copilot Studio を使った場合の突破口**
    Copilot Studio（カスタムエージェント開発環境）を使えば、この壁を**「開発力次第」**で突破できます。標準の「ナレッジ追加（SharePoint指定）」機能を使うだけではテキスト抽出のまま（画像は欠落）ですが、Copilot Studio の「カスタムアクション」機能を使えば、「Graph API を叩いてPDFファイルをダウンロードし、画像としてAzure OpenAI (GPT-4o) に渡して解析させる」という自前の自動パイプライン（Power Automate や外部API）を作成し、Copilot のスキルとして持たせることが可能です。完全に自動化されたマルチモーダルRAGを実現するには、こうしたカスタム開発を行うか、LangChain等で独自のオーケストレータを構築するしかありません。

## ワークフロー（大容量・マルチモーダル対応版）
「画像やレイアウトが欠落する」という生チャンクの弱点を克服し、大容量コンテキスト（200K〜500K）をフル活用するためのワークフローです。システムが裏側で自動的に元ファイルを拾い上げる設計とします。

1. **SSOT整備**: `source_id, file_uri, version, timestamp, trust_score` などを付与し、元ファイル（PDF等）の保存場所をシステムが常に追跡できるようにする。
2. **インデックス生成**: 検索用の「文字ベースの生チャンク（1000〜2000 token）」とメタデータを生成し、ベクトルDBに登録する。
3. **候補抽出（広域検索）**: ユーザーの質問に対し、まずはベクトルDBからテキスト検索で広範に生チャンクを抽出する（`top_k=50〜100`）。
4. **フルソースの自動フェッチ（オートローディング）**: 
   - 抽出された生チャンクのメタデータ（`file_uri`）を参照し、システムが裏側で**元のSSOTファイル全体（画像や図表を含むオリジナルのPDFなど）をストレージから自動的に取得**する。
   - ※テキストのみで十分な軽量クエリの場合はこのステップをスキップし、生チャンクのみを渡す（動的切り替え）。
5. **プロンプト合成とキャッシュ化**: 取得したフルソース（または生チャンク群）をプロンプトの先頭に配置し、「プロンプトキャッシュ」の対象とする。続いてユーザーの入力やチャット履歴を合算しLLMに投入する。
6. **生成と引用（Citation）の強制**: マルチモーダルLLMがフルソース（画像やレイアウト含む）を解析し、回答を生成。末尾に「どのファイルの、どのページ・図表を参照したか」を厳密に引用させる。
7. **SSOT監査ログ**: 生成に使われたファイル履歴をログに残し、情報の鮮度や利用頻度を追跡する。

## チェックリスト（導入時）
- 使用モデルの公式 `context window` を確認し SSOT表と突合する。  
- チャンク長・重複ルール（1000〜2000 tokenなど、大きめのサイズ）をプロジェクトで固定。  
- プロンプトキャッシュ（Prompt Caching）を有効化できるAPI構成になっているか確認する。  
- 出典表示フォーマットと、LLMによる引用（Citation）指示をシステムプロンプトで標準化する。

## 付録: 実務試算例（2026年の目安）

代表的なレンジとして、最新の推奨値である「標準（入力200K/出力200K）」と「高性能（入力500K/出力500K）」の2クラスで設計例を示します。文字数は 1 token $\approx$ 4文字として計算した理論上の最大目安です。

**標準モデル（入力200K / 出力200K 想定）**
   - 合計コンテキスト: 400,000 tokens（約160万文字）
   - 推奨プロンプト予算（安全マージンを含む50〜70%）：200,000〜280,000 tokens（約80万〜112万文字）
   - 構成案（例）: システム指示 5,000 tokens、チャット履歴 5,000 tokens、凝縮要約 20,000 tokens、生チャンク 150,000 tokens、出力予備 20,000 tokens → 合計 200,000 tokens（約80万文字、調整可）
   - 説明: 本書では標準モデル想定を基準に設計例を示す。実運用では各項目を減らして上限内（200K〜280K）に収める。

**高性能モデル（入力500K / 出力500K 想定）**
   - 合計コンテキスト: 1,000,000 tokens（約400万文字）
   - 推奨プロンプト予算（安全マージンを含む50〜70%）：500,000〜700,000 tokens（約200万〜280万文字）
   - 構成案（例）: システム指示 10,000 tokens、チャット履歴 10,000 tokens、凝縮要約 50,000 tokens、生チャンク 400,000 tokens、出力予備 30,000 tokens → 合計 500,000 tokens（約200万文字）
   - 説明: 大規模なドキュメント群の横断検索や、プロジェクト全体のコードベース解析など、より広範なコンテキスト注入が技術的に可能。

注記: 上記はあくまで設計上の目安です。実運用では応答品質・レイテンシ・コスト・検証負荷を考慮し、コンテキスト使用率は50%前後を目安に設計することを推奨します。
