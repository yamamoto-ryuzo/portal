# 第2章 ソースをSSOT（Single Source of Truth）とする設計

## 概要と目的
本章では、RAG（Retrieval-Augmented Generation）において、すべての知識源を一意の**SSOT（Single Source of Truth：信頼できる唯一の情報源）**として管理・設計する方針を示す。

検索ヒット率を上げるためにRAGソース（生データ）をただ無秩序に増やすアプローチは非現実的である。プロンプトに詰め込める情報量は「モデルの最大文字数（トークン数）」で制限されており、不要なデータまで読み込ませることは、コストと遅延（レイテンシ）を増大させ、回答精度を低下させる。

本設計の目的は、コンテキスト制約を論理的に管理し、最もコスト対効果が高く高精度な「SSOTを原典とする派生物（チャンク・要約・埋め込み）の階層制御」を確立することにある。

---

## 章の要点（本質と判断基準）

- **本章の本質**：RAGソースをSSOTとして厳密に管理し、出典追跡（プロヴェナンス）と監査が可能なRAG運用を実現すること。
- **コンテキストは「合算で考える」**：オーケストレーション、RAG参照（要約・生チャンク）、添付ファイル、チャット履歴、プロンプト指示（＋出力予約領域）を合計した文字数をモデルのコンテキスト容量と照合する。
- **巨大なRAGソースの非現実性**：ソースを無尽蔵に増やしても、モデルが処理できるコンテキスト制限により、結果として不要なデータを読み込むことになり、コスト・レイテンシ・検証負荷が急増し、逆に検索精度を損なう。
- **判定式（例）**：
  $$total\_context\_chars \approx orchestration\_chars + RAG\_references\_chars + file\_chars + chat\_history\_chars + prompt\_instructions\_chars + output\_reserved\_chars$$
  
  *【実務試算例】*
  オーケストレーション 500文字 + RAG参照 10,000文字 + ファイル抜粋 2,000文字 + チャット履歴 1,000文字 + 指示 400文字 = **合計 13,900文字**
  
- **結論（改定）**：モデルのコンテキスト容量はRAG運用の現実性を直接左右する。小容量（~4K〜8K）では大量の生チャンクをプロンプトに詰め込む手法は現実的でないが、コンテキストが32K〜128K程度に達する中規模モデル（本書では `GPT-5 Mini` を現行標準の代表として想定）では、いわゆる「仮想的ファインチューニング（インコンテキストでの大量読み込み）」は技術的に実行可能な範囲に入る。

- **注意点**：実行可能性が高まる一方で、次の制約は残る。
   - コスト：大量トークンを毎回供給するとAPIコストが大幅に増加する。\
   - レイテンシ：応答遅延が増えるため対話UXやリアルタイム性が影響を受ける。\
   - 検証負荷：出典検証・出力の正確性確認に必要な工数が増える。

### GPT-5 Mini を標準とした設計上の影響（仮想的ファインチューニングの実用性）

- 想定コンテキスト幅: 本節では保守的に「32K〜128K トークン」を想定する（ベンダー公開値を必ず確認すること）。
- 目安計算: 13,900文字は約3,475トークン（1 token ≒ 4文字換算）に相当し、32Kトークンモデルの総容量から見れば小さい負荷である。したがって、`GPT-5 Mini` を用いる環境では、短中規模のドキュメント群をインコンテキストで与える「仮想ファインチューニング」は現実味を帯びる。

- 実務的おすすめパターン（GPT-5 Mini 標準時）:
   1. 凝縮優先（Summary-first）を継続し、必要な場合のみピンポイントで生チャンクを付与する。
   2. 長文コンテキストを分割して『ロールングコンテキスト（context rolling）』を実装し、段階的に情報を注入する。\
   3. スニペットやfew-shot例をシステム指示で保持し、スタイル/出典ルールを固定する（仮想的FTと併用）。
   4. 頻繁に使う固定挙動（文体、出典出力フォーマット）はLoRA等の軽量FTで補助し、インコンテキストは動的な知識提示に集中する。\
   5. コスト監視・キャッシュ戦略を必須化し、同一問合せでのトークン供給を最小化する。

- 要約: `GPT-5 Mini` 的な中規模コンテキストを標準とすることで「仮想的ファインチューニング」は確かに使える手法になるが、完全自動でSSOTを置き換えるものではなく、引き続きSSOTの整備・差分制御・出典監査が不可欠である。

## 最優先事項：SSOTの抽出と作成

現状で最も重視すべきなのは、無秩序なRAGの拡張ではなく、**「SSOTの抽出と作成（データ整備）」**を最優先することである。SSOTが美しく整備されて初めて、出典保証、確実な差分インジェスト、凝縮要約が可能となり、RAGや正式なファインチューニング（LoRA等）が真価を発揮する。

### 今すぐ実行する項目：
1. **メタ情報の整備**：各SSOTに `source_id`, `version`, `timestamp`, `trust_score` を完全付与。
2. **原文の正規化とチャンク化**：チャンク長（500〜800 token）と重複幅を厳密に定義。
3. **凝縮要約（Summary）の自動生成**：長期メモリ用途として、100〜300 tokenの要約インデックスを自動生成する。
4. **埋め込み・差分インジェスト**：SSOTの更新を検知し、ベクトルDBへ自動同期するCI/Hookを構築。
5. **監査ログと承認フロー**：出典検証履歴と、原典更新時の承認ワークフローを確立。

## 推奨の選定順（SSOTの確実性を最重視）

大規模で未整備な情報源から完璧なSSOTを自動抽出するRAG（検索拡張生成）は技術的難易度が高い。そのため、**「ユーザーが確実なSSOTを直接示す」**ことが、現状最も信頼性の高いアプローチとなる。

1. **AI + 正式なファインチューニング（LoRA等） + ユーザーによるSSOTの直接添付（最良）**
   - *理由*：モデルの振る舞い（出典形式、文体）を正式なファインチューニングで完全に制御し、情報源はユーザーが保証したSSOTに限定する。信頼性と品質の頂点。RAG（検索拡張生成）を併用する場合も、補助的な関連情報検索に留めるべきである。
2. **AI + ユーザーによるSSOTの直接添付（ファインチューニングなし）**
   - *理由*：正式なファインチューニングが利用できない場合の次善策。モデルの振る舞いはプロンプトで制御する必要があるが、情報源が保証されているため信頼性は極めて高い。シンプルで導入しやすい。
3. **AI + 正式なファインチューニング（LoRA等） + RAG（検索拡張生成）**
   - *理由*：ユーザーがSSOTを示さない場合の最良の「自動化」アプローチ。RAGソース全体がSSOTとして高度に整備されていることが前提。正式なファインチューニングにより、自動抽出した情報源からでも安定した品質の回答を生成する。
4. **AI + RAG（検索拡張生成）のみ**
   - *理由*：最も手軽な自動化手法だが、情報源の信頼性もモデルの振る舞いも保証されない。プロトタイプや、間違いが許容される用途に限定すべき。
5. **AI + 仮想的なファインチューニング（インコンテキストでの大量読み込み）**
   - *理由*：実験的アプローチ。大量の情報をプロンプトに詰め込むためコスト・遅延が大きく、本番運用には不向き。SSOTが未整備な段階での一時的な精度検証などに用途が限られる。

## 段階的SSOT同定ワークフロー（提案）

ユーザー案：最初に「数十件規模」で候補チャンクを整理・提示し、実際に生成で使われた最小限のRAGソースをユーザーに提示・確認してから当該ソースを読み込み・確定する流れを推奨します。以下は実務手順例です。

1. 準備：凝縮要約インデックスと生チャンクインデックスを用意し、メタ（`source_id, chunk_offset, trust_score`）を付与する。
2. 候補抽出（広域検索）：クエリに対して `top_k=30〜100` 程度を凝縮インデックスから取得し、各候補を短い要約とスニペットで提示する（出典・スコア併記）。
3. ユーザー検査フェーズ：提示した候補リストをユーザーに見せ、信頼できるソース／優先度を選択してもらう（UI：チェックボックス／レビュー）。
4. 確定後の精選フェーズ：ユーザーが選んだソース（通常は5〜15件）について生チャンクをフェッチし、Cross‑encoderで再ランクして最終選定（例: 3〜10件に絞る）。
5. 生成と検証：最小限の確定ソースを用いて回答を生成し、回答末尾に実際に参照した`source_id`一覧とスニペットを添付してユーザーに提示する。必要なら再確認ワークフローにより追加読み込みを行う。
6. SSOT登録：ユーザー検証で承認されたソースをSSOT候補としてメタを更新し、差分インジェスト・承認ログを残す。

利点：
- 出典の可視化とユーザー合意によりSSOTへ近づける。 
- 不要な大規模読み込みを削減しコストを抑制できる。 

注意点：
- ユーザー操作を前提とするためUX設計が重要（迅速な確認手段を用意すること）。
- 初期候補抽出の`top_k`はモデルのコンテキストとコストに応じて調整する。

運用ヒント：
- 自動化：候補提示時にスコア順だけでなくドメイン重み（trust_score）で優先ソートする。 
- キャッシュ：ユーザーが承認したSSOTはキャッシュして再利用、再評価を簡素化する。 
- 監査：承認の履歴・diff を必ず残し、将来のモデル挙動の追跡を可能にする。

## 推論パイプライン（階層検索モデル）

SSOT設計を具現化する推奨ワークフロー。

1. **第一次検索（凝縮優先）**：
   ユーザクエリを投げ、まずは「凝縮要約」インデックスから `top_k=20` を高速検索。
2. **再ランク（Reranker）**：
   Cross‑encoder等を用いて関連度の高い上位（例: 3〜5件）に絞り込む（トークン・文字数予算内に確実に収める）。
3. **生チャンク（Raw）へのアクセス判定**：
   選定要約から `source_id` を特定。信頼度（confidence）が低い、または詳細な原文フェッチが必要な場合のみ、該当部分の生チャンク（500〜800 token）をピンポイントで動的ロードする。
4. **プロンプト合成と生成**：
   システム指示、厳選参照、会話履歴を合算し、コンテキスト上限の50〜70%に収めてモデルに入力。
5. **出典（プロヴェナンス）強制出力**：
   回答末尾に以下の形式で出典を完全保証。
   `【出典】doc_title (source_id) — chunk_offset — スニペット(〜200字) — score — confidence`

## 付録: モデル別コンテキストサイズと実務試算

### モデル別コンテキストウィンドウの比較（2026年6月 暫定目安）

| モデルのカテゴリー | 代表的なモデル例 | コンテキストサイズ（目安） | 実用上の位置付け |
|:---|:---|:---:|:---|
| 小容量 / レガシー | GPT-3.5 系、初期の GPT-4 構成 | ～8K ～ 32K トークン | 単一ドキュメント要約や短い会話、軽量用途向け。 |
| 中規模（現行標準） | GPT-5 Mini, 拡張版 GPT-4 系、Claude 3 系 | 32K ～ 128K トークン | 複数ドキュメントの横断分析、レポート単位の処理に適する主流帯。 |
| 大容量 | GPT-4o 系、Claude 3.5 Sonnet、Gemini Pro 等 | 128K ～ 1M トークン程度 | 書籍数冊〜大規模コレクションの一括解析、詳細なコードベース解析などに対応。 |
| 超長文特化 / 専用 | Gemini Ultra、長文特化モデル | 1M トークン以上 | 動画トランスクリプト全体解析やソースツリー丸ごとの把握など、極めて大規模なデータ処理向け。 |

注: ベンダーによって公開値は異なります。上表は2026年6月時点の一般的な目安であり、最新値は各ベンダーの公式ドキュメントを参照してください。

### 実務試算例（2026年の目安）

代表的なレンジとして「32K」「128K」「1M（超長文）」の3クラスで設計例を示します。

- **中規模モデル（コンテキスト 32K tok 想定）**
   - 理論最大：約 128,000 文字（1 token $\\approx$ 4文字換算）
   - 実務上の安全なプロンプト容量（50〜70%）：**64,000 〜 89,600 文字**
   - 構成案（例）：システム指示 5,000字、チャット履歴 5,000字、凝縮要約 20件（合計約12,000字）、生チャンク 10件（合計約20,000字）→ **合計 約42,000字**。

- **大容量モデル（コンテキスト 128K tok 想定）**
   - 理論最大：約 512,000 文字
   - 実務上の安全なプロンプト容量（50〜70%）：**256,000 〜 358,400 文字**
   - 構成案（例）：システム指示 10,000字、チャット履歴 10,000字、凝縮要約 50件（約30,000字）、生チャンク 50件（約100,000字）→ **合計 約150,000字**。

- **超長文モデル（コンテキスト 1M tok 想定）**
   - 理論最大：約 4,000,000 文字
   - 実務上の安全なプロンプト容量（50〜70%）：**2,000,000 〜 2,800,000 文字**
   - 構成案（例）：システム指示 20,000字、チャット履歴 20,000字、凝縮要約多数（数百件で100k〜数百万字規模）、生チャンクも大規模に許容→ 大規模コレクションの横断分析や全コードベース解析が技術的に可能。

注記: 上記はあくまで設計上の目安です。実運用では応答品質・レイテンシ・コスト・検証負荷を考慮し、コンテキスト使用率は50%前後を目安に設計することを推奨します。
