# 第2章 RAGの基本理論とSSOT（Single Source of Truth）戦略

## 学習データ・ファインチューニング・RAGの違い

LLMが「知識を持つ・使う」方法は大きく3つに分類できる。

```
① 学習データ（プレトレーニング）
   └─ AIがすでに知っている知識
        └─ モデルの重み（パラメータ）に焼き込まれている
             ＝ 質問しなくても最初から持っている背景知識

② ファインチューニング（Fine-tuning）
   └─ 追加学習でAIの知識・振る舞いを調整
        └─ 学習データと同じく、モデル重みとして埋め込む
             ＝ 「その都度」ではなく、再学習のたびに更新が必要

③ RAG（Retrieval-Augmented Generation）
   └─ AIがその都度、外部の知識ベースを参照する
        └─ 質問が来るたびに関連ドキュメントを検索し、コンテキストとして渡す
             ＝ 「記憶している」のではなく「そのとき読む」
```

### 各アプローチの比較

| 区分 | 知識の場所 | 回答の安定性 | 更新方法 | タイミング |
|---|---|---|---|---|
| **学習データ（プレトレーニング）** | モデルの重み（パラメータ） | **高い**（常に同じ知識から回答） | モデルの再学習（月〜年単位） | 学習時に確定 |
| **ファインチューニング(LoRA等)** | モデルの重み（追加調整） | **高い**（追加学習後は安定） | 再ファインチューニング（週〜月単位） | 学習時に確定 |
| **RAG** | 外部の知識ベース（DB・ファイル） | **検索次第**（ヒットしなければ答えられない） | ドキュメントの差し替えのみ（即時） | **推論のたびに参照** |

> **わかりやすい比喩**  
> - 学習データ／ファインチューニング → 「受験勉強で頭に入れた知識」（試験中は安定して引き出せる）  
> - RAG → 「持ち込み可の試験で、その都度参考書を引く」（参考書に載っていない問いには答えられない）

### ファインチューニング（Fine-tuning / LoRA）の詳細

| 項目 | 内容 |
|---|---|
| **仕組み** | 業務データでモデルの重みを再学習し、知識をモデル本体に埋め込む。**LoRA (Low-Rank Adaptation)** はこの重み更新を非常に軽量なアダプターとして追加する最新手法。 |
| **メリット** | AIが「背景知識」として体得しているため、質問の仕方によらず**安定して回答できる**。RAGのような検索のタイムラグやトークン消費がない。 |
| **デメリット** | 学習コスト（GPU・時間・費用）がかかる。知識更新のたびにデータセット化と再学習が必要。 |
| **向いているケース** | 応答フォーマットの固定化、特定の専門分野（法務・技術等）の文脈理解、推論プロセスの型化（CoT学習）。 |

### RAG（Retrieval-Augmented Generation）の詳細

| 項目 | 内容 |
|---|---|
| **仕組み** | モデルの重みは変えず、質問が届くたびに外部知識ベースを検索してコンテキストとして渡す |
| **メリット** | 知識の追加・更新がドキュメントの差し替えだけで完結。根拠となる出典を明示できる |
| **デメリット** | **検索にヒットしなければ答えられない**（組織固有の知識に基づく回答という意味で）。チャンク設計・インデックス品質が直接回答精度を左右する |
| **向いているケース** | 法令・基準・社内規程など**更新頻度が高い知識**を正確に参照させたい場合 |

> **RAGの根本的な制約**  
> RAGは「参照できる文書が存在し、かつ検索でヒットして初めて機能する」仕組みである。  
> 文書が未登録・チャンク分割が不適切・クエリと文書の表現がずれている、いずれの場合も**組織固有の知識に基づく回答はできない**。  
> これが学習データとの最大の違いであり、**文書の整備・チャンク設計・インデックス品質がRAGの品質そのもの**を決定する。  
>  
> （補足）検索ミス時、LLMは自身の学習データで回答しようとするため、意図せず古い情報や誤情報（ハルシネーション）を返すことがある。これを防ぐにはシステムプロンプトで「提供文書にない場合は答えるな」と明示的に指示する。

### 本書がRAGを採用する理由

```
土木事業管理の知識基盤
├─ 建設業法・設計基準・積算基準 → 改訂頻度が高い（学習データには含まれない最新版）
├─ 過去の事業実績・工事記録   → 蓄積・更新が継続的に発生（組織固有の情報）
└─ 社内規程・通達・Q&A       → 組織ごとに異なり個別管理が必要（非公開情報）
```

上記はいずれも**AIがもともと知らない・知れない知識**であり、「モデルに焼き込む」より  
「その都度参照させる」ほうが合理的である。  
**本書の主軸はRAGによるリアルタイム知識参照**に置く。

### ファインチューニングとRAG、正確性はどちらが高いか

**検索がヒットした場合、事実の正確性はRAGの方が高い。** これは直感に反するが、理由は明確である。

| 観点 | ファインチューニング | RAG |
|---|---|---|
| **知識の保持方法** | 学習データを重み（パラメータ）に**圧縮して記憶** | 元文書を**そのまま**LLMに渡す |
| **正確性の問題** | 圧縮の過程で細部が歪む・複数知識が混合する → ハルシネーションの原因になる | 元文書が正確であれば、その内容をそのまま参照できる |
| **ヒットしない場合** | 不正確でも「何かしら答える」 | 組織固有の知識は答えられない |

> **比喩で整理**  
> - ファインチューニング = 講義ノートを暗記する（要点は覚えるが、細かい数字や条文は記憶が歪みやすい）  
> - RAG = 試験中に原文テキストを参照する（見ている文書が正しければ、写し間違えない限り正確）

ファインチューニングが「安定して回答できる」のは事実だが、**知識の細部（法令条文・数値・固有名詞）の正確性においてはRAGが勝る**。これが土木事業管理のような精度要求が高い業務でRAGを主軸とする根拠の一つである。

### RAGパイプラインは「取りこぼしが少ない」のか

**RAGはファインチューニングではなく、検索システムである。** LLMのパラメータは一切変わらない。

RAGが「取りこぼしが少ない」ように見える理由は、**検索ミス時にLLMが事前学習済み知識で回答しようとするから**であり、これはRAGの機能ではなくLLM自体の挙動である。

```
RAGパイプラインの回答フロー（正確な理解）

質問
 │
 ├─ 外部ベクトルDBを検索
 │    ├─ ヒット → 検索結果をコンテキストとしてLLMに渡す
 │    │            → LLMは渡された文書を参照して回答（高精度）
 │    │
 │    └─ ミス  → コンテキストに何も渡さない
 │                → LLMは事前学習済み知識だけで回答（RAGは無関係）
 │
 └─ 注意：「検索ミス時の回答」はRAGの機能ではなくLLM本来の挙動
```

> **「取りこぼしが少ない」は諸刃の剣**  
> 組織固有の質問（自社規程・個別工事記録など）で**検索ミスが起きたとき、LLMは事前学習済み知識で「それらしい回答」を生成しようとする**。  
> これが古い情報・一般論・架空情報（ハルシネーション）として出力される原因である。  
>  
> RAG_Systemが「答えてしまう」のは検索が成功したからではなく、**LLMが検索ミスに気づかず回答するから**である。  
> これを防ぐにはシステムプロンプトで **「提供文書にない場合は回答しない」と明示指示する**。

### ハルシネーション対策：「答えるな」より「情報源を推論で分離する」

「提供文書にない場合は答えるな」はシンプルだが、汎用的な質問（用語の定義・一般的な手続きなど）まで遮断してしまう過剰制御になりやすい。より実用的な設計は、**回答の情報源をLLMが自己推論で分類し、明示したうえで答える**ことである。

| 状況 | 情報源 | 扱い方 |
|---|---|---|
| RAGにヒット | **SSOT**（組織固有文書） | 出典付きで回答。これが唯一の正解（Single Source of Truth） |
| RAGにミス・汎用質問 | LLMの学習データ（一般論） | 「一般的な情報として」と明示したうえで回答 |
| どちらか不明 | 不明 | 推論で分類し「組織文書に根拠なし」とユーザーに提示 |

```
回答の信頼性を情報源で3分類する設計

質問
 │
 ├─ RAGヒット ──────→ [SSOT] 「▶ 規程第○条に基づく回答」（高信頼・出典明示）
 │
 ├─ RAGミス＋汎用質問 → [一般論] 「▶ 一般的な解釈として…（組織文書に根拠なし）」
 │
 └─ 判断困難 ─────→ [要確認] 「▶ 組織固有の情報が見当たりません。ご確認ください」
```

この設計の利点：
- **ハルシネーション = 「組織固有の質問に対してSSOTなしで答えること」** と定義できる
- 一般論はハルシネーションではなく**意図的な補完**として扱える（取りこぼしが減る）
- ユーザーが「これはRAGからの回答か、LLMの推測か」を判断できる

> **実装上のポイント**  
> システムプロンプトに「検索結果が存在する場合は必ず出典を示せ。存在しない場合は『組織文書に根拠なし』と前置きして一般論を述べよ」と指示することで、  
> LLMが回答前に情報源を自己分類する動作を誘導できる。  
> マルチエージェント構成ではこの「情報源の判定」を専用エージェント（検証エージェント）に委ねることで精度をさらに高められる（詳細は7章）。

> **ファインチューニングとRAGは独立した技術であり、組み合わせることもできる**  
> ファインチューニング済みモデルをRAGパイプラインのLLMとして使うことは可能だが、  
> それぞれの役割は変わらない。  
> - ファインチューニング → モデルのパラメータを変える（学習）  
> - RAG → クエリ時に文書を検索してコンテキストに渡す（検索）  
>  
> 組み合わせても「RAGが検索システムである」という本質は変わらない。  
> RAGの品質を上げたいなら「チャンク設計・インデックス精度・検索戦略」を改善する。本書の構成はそのまま流用できる。



## 2.1 RAG拡張による回答品質の向上

#### RAGの根本的な動作原理：AIは「全件」ではなく「検索された数件」しか見ない

単純な1回検索型のRAGでは、1000件のソースファイルを登録しても、AIが1回の回答で主に使うのは**その中から検索で取り出された上位N件（例：10件）だけ**である。  
登録件数を増やしても、1回の回答ラウンドでLLMに直接渡される文書数は通常は増えない。

```
1000件のソースファイルをRAGに登録した場合の動作

1000件 → ベクトルDB（インデックス）に格納
                    │
              質問が来る
                    │
              ベクトル検索（全1000件を対象）
                    │
            上位10件を取り出す  ← ここで選別が起きる
                    │
     ┌──────────────┘
     │  取り出された10件のテキスト
     │  ＋ 質問
     └──────────────→ LLM が回答

残り990件は、その回答ラウンドでは通常は直接参照されない
```

| よくある誤解 | 正しい理解 |
|---|---|
| 1000件の知識を持つAIができる | 1000件から上位N件を選ぶ検索システムができる |
| 登録すれば全部使ってくれる | 検索でヒットした分しか使わない |
| RAGが賢くなる | LLM自体は変わらず、検索とコンテキスト供給の質が改善される |
| 登録件数を増やせば回答が良くなる | 正しい文書が上位N件に入れる設計になっているかが重要 |

> **これが②インデックス設計が重要な理由**  
> 「この質問に答えるべき文書」が上位N件に含まれるかどうかは、チャンク分割・埋め込みモデル・検索戦略の設計次第である。  
> どれだけ優れた文書を1000件登録しても、**検索でその文書が選ばれなければ、その回答ラウンドでは使われない。**  
> RAGの品質改善とは、多くの場合「正しい文書を上位N件に入れる確率を上げる」ことである。

---

RAGの回答品質は「どのシステムを使うか」よりも「何を・どう登録し・どう検索するか」が決定的に重要である。  
品質を左右する要素を3層に整理し、各層での改善戦略を示す。

```
RAG回答品質の決定構造

① 知識源の整備（What）
   └─ 文書が存在しなければ始まらない。これが最優先の前提条件。
        ─ 登録すべき文書：法令・基準の最新版、社内規程、過去事例、Q&A
        ─ 整備の視点：網羅性（抜け漏れなし）・鮮度（最新版かどうか）・精度（誤記なし）

② インデックス設計（How to store）
   └─ 文書があっても「検索でヒットしない形」に分割されると品質が落ちる。
        ─ チャンクサイズ：小さすぎると文脈が切れる、大きすぎるとノイズが増える
        ─ オーバーラップ：文の境界でチャンクが切れる問題を緩和（50〜100トークン推奨）
        ─ 分割単位：段落・見出し単位が意味的まとまりを保ちやすい

③ 検索システムの選択（How to retrieve）
   └─ ②を制御できるシステムを選ぶことが③の選定基準になる。
        ─ ベクトル検索：意味的類似性でヒット（キーワード一致不要）
        ─ ハイブリッド検索：ベクトル＋キーワードの併用（抜け漏れ低減）
        ─ 再ランキング：Cross-Encoderで上位N件を再順位付け（精度向上）
        ─ 取得件数の制御：top_k を質問の複雑さに応じて調整
```

#### 制御可能性による品質改善の限界

| システム | ①知識源 | ②インデックス | ③検索戦略 | 改善余地 |
|---|:---:|:---:|:---:|---|
| **Azure AI Search（AutoGen接続）** | ◎ユーザー管理 | ◎ユーザー制御 | ◎ユーザー制御 | **最大**（全層を改善できる） |
| **Dify** | ◎ユーザー管理 | ○GUI設定 | ○GUI設定 | **大**（PoC段階では十分） |
| **LlamaIndex** | ◎ユーザー管理 | ◎コードで制御 | ◎コードで制御 | **最大** |
| **M365 Copilot / Microsoft Graph検索** | △SharePoint依存 | ✗ユーザー非制御 | ✗ユーザー非制御（内部実装依存） | **極小**（主に①のみ改善可能） |
| **Box API** | △Box内文書のみ | ✗非制御 | ✗非制御 | **極小** |

> **M365 Copilot の制約をどう扱うか**  
> M365 Copilot では、少なくとも通常の利用者・管理者の立場からは②③を直接制御しにくく、「チャンク設計を変える」「取得件数を明示的に増やす」といった品質改善手段は取りにくい。  
> そのため、SharePoint上の文書整備（①）が主要な改善レバーになる。  
> 精度要求が高い業務（法令照会・積算根拠確認等）には、②③を制御できる **Track B（AutoGen + Azure AI Search）** を採用し、  
> M365 CopilotはUI統合・社内周知のフロントエンドとして位置づける構成が実用的である。

#### Box AI Hubs と M365 Copilot の違い

Box AI Hubs では、ユーザーがHubに紐づけた複数ファイル群を**参照候補の集合**としてまとめて扱える。  
この意味で、AIが参照しうる範囲を「1ファイル単位」ではなく「Hubで指定した複数ファイル群」まで広げることはできる。  
ただし、M365 Copilot と同様に、**毎回その複数ファイル全体をLLMが丸ごと読むわけではなく、実際には質問に応じて取得された一部の内容だけが回答に使われる**と理解するのが正確である。

| 観点 | Box AI Hubs | M365 Copilot |
|---|---|---|
| **参照範囲の指定** | Hubに関連文書群をまとめて指定できる | 主にM365全体の権限内コンテンツから検索される |
| **ユーザーの感覚としての対象範囲** | 「このHub内の文書群を見てほしい」と範囲を絞りやすい | SharePoint / OneDrive / Teams 等の広い範囲から暗黙に探索される |
| **実際の回答時の動作** | Hub内の全ファイルを常時読むのではなく、関連部分が取得されて使われる | 候補群から関連部分が内部的に取得されて使われる |
| **取得件数の制御** | 内部実装依存で、細かな制御は限定的 | 内部実装依存で、細かな制御は限定的 |
| **実務上の意味** | 参照候補の集合を業務単位・案件単位で整理しやすい | Microsoft 365 全体の業務文書を横断利用しやすい |

> **要点**  
> Box AI Hubs の利点は、「AIが全ファイルを読むこと」ではなく、**参照候補の母集団をHub単位で整理・限定しやすいこと**にある。  
> したがって、「思考対象ファイルを複数ファイルに拡大できる」という理解は半分正しく、より正確には **「検索対象・参照候補をHub内の複数ファイル群に広げられる」** である。

#### 大量知識を扱うときの選択基準：RAG強化・A2A・MCP・ファインチューニングの役割分担

今後のツール選定で最も重要なのは、**「1回の回答でAIに渡せる知識量には上限がある」** という前提から出発することである。  
多くのRAG_Systemでは、登録済み文書が1000件あっても、1回の回答でLLMに直接渡るのは上位数件〜十数件の**チャンク**であり、文書全体ではない。  
したがって、知識量が増えたときに考えるべき順番は「全部を一度に見せる方法」ではなく、**必要な知識を段階的に正しく取り出す方法**である。

#### まず区別すべき5つの選択肢

| 手法 | 何を増やすか | 向いている課題 | 限界 |
|---|---|---|---|
| **単純RAG** | 1回の検索で取る上位チャンク | FAQ、単一文書照会、短い規程確認 | 1回で拾えない情報は使えない |
| **RAG強化** | 検索精度と取得結果の質 | 文書数が多いが、検索設計で対応可能な課題 | 1回の文脈上限自体は残る |
| **A2A / 再帰的探索** | 検索回数と検討ステップ | 複数文書横断、比較、段階的調査、案件別検討 | 構成が複雑になり、実装・運用コストが増える |
| **MCP** | 外部ツール・外部知識源への接続性 | Box、SharePoint、DB、社内API、検索APIを統一的に接続したい課題 | それ自体は検索品質や推論能力を上げない |
| **ファインチューニング** | モデルの振る舞い・出力の癖 | 定型応答、分類、書式統一、社内文体の固定 | 最新知識の大量追加には不向き |

#### MCPは何が違うのか

MCP（Model Context Protocol）は、RAG・A2A・ファインチューニングとは役割の層が異なる。  
RAG、A2A、ファインチューニングは「AIがどう知識を使うか」の方式だが、MCPは**AIがどの外部ツールや外部データ源にどう接続するか**を標準化するためのプロトコルである。

| 概念 | 役割 | 何を解決するか |
|---|---|---|
| **RAG** | 検索して知識を渡す | 外部文書から必要部分を取り出して回答に使う |
| **A2A** | エージェント間の分業・段階探索 | 1回で足りない調査や多段の検討を回す |
| **MCP** | 外部接続の標準化 | Box、SharePoint、DB、API、検索ツールを同じ作法で呼ぶ |
| **ファインチューニング** | モデルの振る舞いを学習で固定 | 書式、分類、応答傾向を安定化する |

したがって、MCPそのものは知識量を増やさず、検索精度も直接は改善しない。  
MCPは、**RAGやA2Aが利用するための外部接続の配線**を標準化する層だと理解するのが正確である。

```
実務での関係

AI
 │
 ├─ MCPで外部サービスへ接続する
 │    ├─ Box
 │    ├─ SharePoint
 │    ├─ 社内DB
 │    └─ 検索API
 │
 ├─ その先でRAG検索を行う
 │
 ├─ 必要ならA2Aで追加探索・検証を行う
 │
 └─ 最終的に回答する
```

#### MCPを選ぶべき場面

MCPを重視すべきなのは、「知識の使い方」よりも「知識源へのつなぎ方」がボトルネックになる場面である。  
例えば、次のようなケースではMCP対応が重要になる。

| 状況 | MCPが重要な理由 |
|---|---|
| Box、M365、社内DB、独自APIを横断接続したい | 接続方式を個別実装せずに揃えやすい |
| 接続先の追加・差し替えを頻繁に行いたい | エージェント側の変更を小さくしやすい |
| フレームワークをまたいで同じツールを使いたい | ツール定義の再利用性が高まる |
| 将来の拡張性を重視したい | ベンダー固有結合を減らしやすい |

逆に、「MCP対応だからRAG品質が高い」とは言えない。  
MCPで優れていても、検索システムの設計が弱ければRAG品質は上がらない。  
本書の評価軸で言えば、**MCPは接続性の軸、RAGは検索品質の軸、A2Aは探索・分業の軸**である。

#### マルチエージェントはどの層に当たるか

マルチエージェントは、RAG・MCP・ファインチューニングのどれか1つを指す言葉ではなく、**複数のエージェントが役割分担して処理を進めるオーケストレーション構造**を指す。  
本書の文脈では、マルチエージェントは主に **A2A / 再帰的探索の層** に位置づく。  
つまり、RAGは「文書を取る仕組み」、MCPは「外部に接続する仕組み」、ファインチューニングは「振る舞いを固定する仕組み」であり、**マルチエージェントはそれらをどう分担・連携させるかの設計**である。

| 構成要素 | 役割 |
|---|---|
| **RAG** | 必要な文書断片を検索して各エージェントに渡す |
| **MCP** | 各エージェントがBox、SharePoint、DB、API等に接続する |
| **マルチエージェント** | 受付、検索、検証、統合などの役割を分担する |
| **ファインチューニング** | 各エージェントの出力様式や判断傾向を安定化する |

例えば、次のような流れが典型的なマルチエージェント構成である。

```
ユーザー質問
  │
  ├─ 受付エージェント：質問を分解する
  ├─ 検索エージェント：RAGで候補文書を探す
  ├─ 検証エージェント：根拠の妥当性を確認する
  └─ 統合エージェント：最終回答を組み立てる
```

このとき、A2Aはエージェント同士の受け渡し方法、RAGは各エージェントが参照する知識取得手段、MCPはその知識源やツールへの接続手段である。  
したがって、**マルチエージェント = A2Aを含む上位の協調構造**であり、RAGやMCPはその中で使われる部品だと整理すると混乱しにくい。

#### 判断の原則：知識が多いときほど、先にRAGと探索を強くする

知識量が増えたとき、最初に検討すべきはファインチューニングではない。  
理由は、ファインチューニングは**知識の追加先がモデルの重み**であり、更新頻度が高い文書群や案件ごとの差分を柔軟に扱うのに向かないからである。  
一方、RAG強化やA2Aによる再帰的探索は、知識を外部文書のまま保持しつつ、必要な部分だけを段階的に回収できる。

ただし、そのRAGやA2Aが参照する先を増やしたい場合には、MCPが有効になる。  
つまり、MCPはRAGやA2Aの代替ではなく、**それらを外部知識源へ接続するための基盤**である。

```
知識量が増えたときの基本方針

文書が増える
   │
   ├─ まずやること：検索精度を上げる
   │    ├─ チャンク設計
   │    ├─ 埋め込みモデル見直し
   │    ├─ ハイブリッド検索
   │    └─ 再ランキング
   │
   ├─ それでも1回で足りない
   │    └─ 複数回検索・A2A・再帰的探索へ拡張
   │
   └─ なお固定的な応答品質を揃えたい
  └─ その部分だけファインチューニング
```

#### A2A / 再帰的探索が必要になる場面

A2Aやマルチエージェントが有効なのは、「1回の検索結果だけでは判断が完結しない」場面である。  
例えば、次のような課題は単純RAGよりも段階探索に向く。

| 課題 | 単純RAGが弱い理由 | A2A / 再帰的探索が有効な理由 |
|---|---|---|
| 複数規程の整合確認 | 1回の上位チャンクに全規程が揃わない | 観点ごとに追加検索し、比較用の材料を集められる |
| 案件別の過去事例探索 | 類似案件が1回で十分に取れない | 条件を変えながら段階的に事例を掘れる |
| 法令・通達・社内規程の横断確認 | 情報源が分散し、単一検索では漏れやすい | 情報源ごとに専門エージェントを分けて回収できる |
| 根拠確認付きの最終回答 | 検索結果の信頼性検証が必要 | 検索・検証・要約を分業できる |

このときのA2Aは、「最初から全部知っているAI」を作る方法ではなく、**必要な知識を複数回の検索と検証で集めるオーケストレーション**である。

#### ファインチューニングを選ぶべき場面

ファインチューニングが有効なのは、主に**知識量の拡張**ではなく**振る舞いの固定化**である。  
具体的には、次のような用途で使う。

| 向いている用途 | 理由 |
|---|---|
| 回答フォーマットの固定 | 毎回同じ様式で出力させやすい |
| 分類・判定タスクの安定化 | ラベル付けや定型判定の一貫性を上げやすい |
| 社内文体・表現ルールの統一 | 出力スタイルの揺れを抑えやすい |
| 特定タスクの高速化 | 毎回長い指示を与えなくてもよくなる |

逆に、法令改正、社内規程更新、案件資料追加のような**頻繁に変わる知識の追加先**としては、RAGのほうが原則として適している。

#### 実務での選定順序

本書で推奨する選定順序は次の通りである。

1. まず単純RAGで必要文書が上位に入るか確認する
2. 足りなければ、ハイブリッド検索・再ランキング・top_k調整などでRAGを強化する
3. 接続先が増えるなら、MCPで外部知識源やツールの接続を標準化する
4. それでも1回の検索では足りない課題に対して、A2Aや再帰的探索を導入する
5. 最後に、出力形式や判定の安定化が必要な部分だけファインチューニングする

> **設計上の結論**  
> 「AIがRAGで追加知識を使える量には上限がある」ことは事実だが、その対策の第一選択はファインチューニングではない。  
> 第一選択は **RAGの検索品質向上**、接続先が多いなら **MCPで外部接続を整理** し、次に **A2Aによる段階的探索** を入れる。ファインチューニングはその後に **振る舞い固定のために限定的に使う** のが実務的である。

#### 品質改善の優先順位

```
RAG品質が低い場合のチェック順序

Step 1: 知識源の確認（①）
  └─ 必要な文書が登録されているか？ → 登録漏れが最大の原因
  └─ 最新版か？ → 古い文書が混在していないか
  └─ 誤記・フォーマット崩れがないか？ → PDFからの変換ミス等

Step 2: インデックス設計の見直し（②）
  └─ チャンクが大きすぎてノイズが多い → サイズを500〜800トークンに調整
  └─ 文の途中で切れている → オーバーラップを50〜100トークン追加
  └─ 表・箇条書きが壊れている → 前処理でMD変換してから登録

Step 3: 検索戦略の強化（③）
  └─ キーワードが一致しないとヒットしない → ハイブリッド検索に切替
  └─ 上位に無関係な文書が来る → Cross-Encoderで再ランキング
  └─ 取得件数が少なく文脈が不足 → top_k を5〜10に増加
  └─ 埋め込みモデルがドメインに合わない → BGE等の日本語特化モデルを試す
```

> **「モデルを変えても精度が上がらない」理由**  
> RAG_Systemの精度が低い場合、LLMモデルのアップグレードではなく上記 Step 1〜3 の順に確認することが効率的である。  
> 多くの場合、原因は「文書の未登録（Step 1）」または「チャンク設計（Step 2）」にある。  
> モデルを変えることで、検索語の言い換えや取得結果の統合が多少改善することはあるが、インデックスに欲しい情報が入っていなければ改善は限定的である。





## 要旨（先に結論）
この章は、RAG（Retrieval‑Augmented Generation）運用において「SSOT（Single Source of Truth）」を中心に据え、モデルのコンテキスト制約とオーケストレーション設計を一致させることで、コスト・レイテンシ・検証負荷を抑えつつ高精度な生成を実現するための実務方針を示します。

## 本章で使う基本用語（初心者向けガイド）
専門用語が多いため、まずは以下の基本概念を押さえてください。

**システム全般・基本概念**
- **LLM（大規模言語モデル）**: ChatGPT（OpenAI）や Gemini（Google）、Claude（Anthropic）などのAIそのものを指します。
- **RAG（検索拡張生成）**: AI（LLM）が知らない最新情報や社内データを、検索システムを使って探し出し、その回答の「カンペ（情報源）」としてAIに渡して答えさせる仕組みです。
- **RAGソース**: RAGが検索し、AIに渡すためのデータの元となる**「文書ファイル全体」**のことです（例：100ページの社内マニュアルPDF、議事録のWordファイルなど）。
- **SSOT（Single Source of Truth / 唯一の正しい情報源）**: 「これが絶対に正しい」と組織内で合意されたマスターデータのこと。RAGにゴミデータ（古い情報や間違った情報）を渡すとAIも間違えるため、「AIにはSSOTだけを渡す」ことが極めて重要になります。

**データの分割と要約に関する用語**
- **チャンク**: 長い文章（RAGソース）をAIが読みやすいように、数百文字〜数千文字程度の「意味のかたまり」に**切り分けたブロック**のことです。
- **生チャンク（詳細参照）**: RAGソースを切り分けただけで、**中身を要約せず原文そのままのブロック**のことです。詳細な情報が残っていますが、文字数を多く消費します。
- **凝縮要約（インデックス）**: 原文（生チャンク）の内容をAIが短く100〜300文字程度に**要約したブロック**のことです。検索を速く・安く行うための「見出し」や「目次」として使います。

**AIの処理量とプロンプト構成に関する用語**
- **トークン（Token）**: AIがテキストを読み書きする際の「文字の数え方の単位」です。英語なら `1トークン ≒ 約4文字`、日本語なら `1トークン ≒ 約1〜2文字` の目安です。
- **コンテキスト（コンテキストウィンドウ）**: AIが「1回の会話で一度に読める・記憶できる文章量の限界」のことです。例えば「コンテキスト200K（20万トークン）」のAIなら、本数冊分（約80万文字）の資料を一度に読んで回答できます。
- **プロンプトキャッシュ（Prompt Caching）**: 毎回同じ長い資料（SSOT）をAIに読ませるとお金と時間がかかりますが、一度読んだ資料をAI側に「一時記憶（キャッシュ）」させておくことで、2回目以降の回答を安く・爆速にする最新技術です。
- **システム指示（システムプロンプト）**: 「あなたは優秀なアシスタントです」「必ず敬語で答えてください」といった、AIの振る舞いやルールを規定する大前提の指示文のことです。
- **チャット履歴**: ユーザーとAIの過去の会話のやりとりのことです。文脈を保つために、直近の会話をAIに毎回一緒に渡す必要があります。
- **出力予備（モデルへ確保するバッファ）**: AIが回答（出力）するための空きスペースです。AIのコンテキスト上限ギリギリまで資料（入力）を詰め込むと、AIが回答を書くための余白がなくなりエラーになるため、意図的に空けておく余裕（バッファ）のことです。

**（参考）一連の処理フロー（言葉がどう使われるか）**
実際のRAGシステムでは、上記の言葉が以下の順番で組み合わさって動きます。
1. **ユーザーの入力**: ユーザーがチャット画面で質問（クエリ）を入力する。
2. **AIのRAG検索（インデックス検索）**: ユーザーの質問に関連しそうなデータを「凝縮要約」の中から高速に探し出す。
3. **AIのRAG抽出（生チャンク取得）**: 検索でヒットした要約の元となっている詳しい原文（生チャンク）をデータベースから取り出す。
4. **元のRAGソースの自動取得（オートローディング）**: 
   - 抽出した生チャンクに記録されている「ファイルの保存場所」の情報を使い、システムが裏側で**「元のRAGソース（画像や図表を含むPDFなどのファイル全体）」**を自動的に引っ張ってきます。
   - （※画像を含まない単純なテキスト質問の場合は、生チャンクだけをAIに渡すことで処理を軽くすることもあります）
5. **プロンプト合成**: AIに渡すための最終的な「カンペ」を作る。[システム指示] ＋ [引っ張ってきたRAGソース全体（SSOT）] ＋ [チャット履歴] ＋ [ユーザーの入力] を、コンテキスト上限を超えないように1つの文章に合体させる。
6. **AIの回答**: 合体されたカンペを読んだLLMが、出力予備（バッファ）を使って最終的な答えを生成し、画面に返す。

## 背景と目的
・RAGソースは量だけでなく「整備されたSSOT」を起点とすることが品質と検証性を担保する最短ルートです。  
・モデルのコンテキスト上限（SSOT参照）を踏まえ、オーケストレーション側でチャンク化・再ランク・段階注入を行うことが本設計の核です。

## コンテキスト予算の判定式（設計式）
設計上の合算式（トークン換算）:
$$total\_context\_tokens \approx orchestration\_tokens + RAG\_references\_tokens + file\_tokens + chat\_history\_tokens + prompt\_instructions\_tokens + output\_reserved\_tokens$$

実務試算例（標準モデル想定 — 入力200K / 出力200K）: 合計コンテキスト 400K トークンのうち、プロンプト予算は安全マージンを考慮して50〜70%を推奨（= 200K〜280K トークン）。
※文字数換算の目安：英語主体なら 1 token $\approx$ 約4文字、日本語主体なら 1 token $\approx$ 約1〜2文字。以下は「1 token $\approx$ 4文字換算」での最大目安字数を併記します。

例（概念）: オーケストレーション 500 tokens (約2,000字) + RAG参照 10,000 tokens (約40,000字) + ファイル抜粋 2,000 tokens (約8,000字) + チャット履歴 1,000 tokens (約4,000字) + 指示 400 tokens (約1,600字) + 出力予備 5,000 tokens (約20,000字) = 合計 18,900 tokens（約75,600文字、標準予算の約9.5%）。

設計方針: 使用モデルのSSOTに基づき、合計がモデルコンテキストの50〜70%以内に収まるよう予算配分する。ここでは標準想定として「入力200K / 出力200K」を前提に設計例を示す。

**標準モデル（入力200K / 出力200K 想定） — 推奨プロンプト配分（例）**:  
- システム指示: 1,000〜5,000 tokens（約4,000〜20,000字）  
- 凝縮要約（インデックス）: 合計 10,000〜30,000 tokens（約4万〜12万字。例: 20件×500 tokens）  
- 生チャンク合計（詳細参照）: 20,000〜60,000 tokens（約8万〜24万字）  
- チャット履歴: 1,000〜10,000 tokens（約4,000〜40,000字）  
- 出力予備（モデルへ確保するバッファ）: 5,000〜20,000 tokens（約2万〜8万字）  
上記合計が 200K〜280K tokens（約80万〜112万文字）の範囲に収まるよう調整してください。

## 影響と設計パターン（大容量コンテキスト前提のSSOT中心設計）
標準（200K）および高性能（500K）という広大なコンテキストウィンドウを前提とする場合、RAGの設計パターンは従来の「極限まで削る」アプローチから「確実なSSOTを広く取り込む」アプローチへシフトします。

1. **Massive Context RAG（広域一括注入）**: 凝縮要約だけでなく、関連する可能性のある生チャンクを数十〜数百件規模で一括してプロンプトに注入します。検索漏れ（False Negative）のリスクを下げ、LLM自身の文脈理解力にフィルタリングを委ねます。
2. **Context rolling（段階注入）**: 500Kを超える極端な長文（例：巨大なコードベース全体や全履歴）の処理には、依然としてロールングで段階的に情報を注入する設計が必要です。
3. **Few‑shot とシステム固定文**: スタイルや出力フォーマットの指定、大量のFew-shot例をシステム指示に含めてもコンテキストを圧迫しません。頻出挙動はプロンプトで十分に制御可能であり、軽量FT（LoRA等）の必要性は低下します。
4. **キャッシュと差分インジェスト**: 広大なコンテキストを毎回フルロードするとコストと遅延が増大します。承認済みSSOTはプロンプトキャッシュ（Prompt Caching）を活用し、更新は差分インジェストで同期します。
5. **コスト監視とバジェット**: 巨大コンテキストモデルは入力トークン単価の管理が重要です。プロンプトキャッシュのヒット率を最大化するよう、静的（不変）なSSOTブロックと動的（クエリ依存）なブロックを分離するプロンプト構成が必須です。

### （重要）テキストRAGの限界と「画像・図表の欠落」問題
M365のMicrosoft Graphや一般的なベクトルDBを用いたRAG検索は、**純粋な「テキストデータ（文字情報）」のみ**を抽出します。
そのため、LLM（回答AI）自身が GPT-4o や Gemini などのように「画像を理解できる能力（マルチモーダル）」を持っていたとしても、**AIに渡されるカンペ（生チャンク）の中にそもそも画像が存在しないため、画像やグラフに基づいた回答は不可能**になります。

**解決策のアプローチ：**
- **テキストRAG（通常）**: 文字ベースの検索に割り切る。画像内の情報は事前OCRや図表要約AIを使って「文字として」チャンクに埋め込んでおく。
- **フルソース読込（マルチモーダル・広域文脈対応）**: もし画像やレイアウトの理解が必須なクエリの場合は、RAGによる「生チャンク」の抽出で済ませるのではなく、**ヒットしたRAGソース（元のPDFやWordファイル全体）を丸ごとAIのコンテキストに投げ込む（再読込する）**必要があります。
  - *自動化の理想*: システム側で「検索でヒットした生チャンクのメタデータ（ファイルパス等）」を読み取り、**裏側で自動的に元のファイル全体をフェッチしてLLMに渡す自動パイプライン**を独自構築することが可能です。大容量コンテキスト（200K〜500K）の時代だからこそ実現できる強力なアーキテクチャです。
  - *【重要】マルチモーダル不要（テキストのみ）のタスクにおける優位性*: 画像理解が不要な純粋なテキスト抽出であれば通常のテキストRAGと結果は同じですが、フルソース読込にはそれを超える大きなメリットがあります。第一に、チャンク分割では失われる**「文書全体を俯瞰した文脈理解（例：資料全体の要約、離れた章同士の関連性分析）」**が可能になります。第二に、ベクトル検索特有の**「検索漏れ（False Negative）」**が発生した場合でも、フルソースがプロンプト内にあるためLLM自身の高い推論能力で必要な箇所を見つけ出すことができ、回答の確実性が飛躍的に向上します。
  - *【M365 Copilotの現実と構造的限界】*: 
    現在の標準的な M365 Copilot（Graph Grounding）は、セキュリティと速度を優先したSaaS製品であり、以下の構造的な制約（ブラックボックス）を抱えています。
    1. **Graph APIが返すのは「テキストのみ」**: Microsoft GraphがSharePoint等から検索してLLMに渡すのは「テキスト化された生チャンクの文字列」と「ファイルのURL」だけです。画像やレイアウトのバイナリデータは渡されません。
    2. **LLMはファイルに直接アクセスできない**: 「ファイルのURL」をLLMに渡しても、LLM自身はWebブラウザのように社内のURLにアクセスしてPDFをダウンロードする機能や権限を持っていません。
    3. **オートローディング（自動再読込）のパイプラインがない**: M365 Copilotの内部には「URLを受け取って、裏でこっそり元ファイルをダウンロードし、画像に変換してLLMに投げ直す」という仕組みが実装されていません。ユーザーや管理者がこの内部仕様を改造することも不可能です。
    
    **結論として、M365 Copilotを使う限り、裏で動くモデルがどれほど優秀なマルチモーダルAI（GPT-4o等）になっても、RAGとして画像を理解して回答することはできません。**
    これを解決するには、「ユーザーが手動でチャット画面に該当ファイルを直接添付（参照指定）する」か、「事前にWordやPowerPoint側で図表のAltテキスト（代替テキスト）を詳細に書き込んでおく（テキストRAGへの歩み寄り）」というアナログな運用カバーが必須となります。
    
    **※補足：M365 Copilot Studio を使った場合の突破口**
    Copilot Studio（カスタムエージェント開発環境）を使えば、この壁を**「開発力次第」**で突破できます。標準の「ナレッジ追加（SharePoint指定）」機能を使うだけではテキスト抽出のまま（画像は欠落）ですが、Copilot Studio の「カスタムアクション」機能を使えば、「Graph API を叩いてPDFファイルをダウンロードし、画像としてAzure OpenAI (GPT-4o) に渡して解析させる」という自前の自動パイプライン（Power Automate や外部API）を作成し、Copilot のスキルとして持たせることが可能です。完全に自動化されたマルチモーダルRAGを実現するには、こうしたカスタム開発を行うか、LangChain等で独自のオーケストレータを構築するしかありません。

## ワークフロー（大容量・マルチモーダル対応版）
「画像やレイアウトが欠落する」という生チャンクの弱点を克服し、大容量コンテキスト（200K〜500K）をフル活用するためのワークフローです。システムが裏側で自動的に元ファイルを拾い上げる設計とします。

1. **SSOT整備**: `source_id, file_uri, version, timestamp, trust_score` などを付与し、元ファイル（PDF等）の保存場所をシステムが常に追跡できるようにする。
2. **インデックス生成**: 検索用の「文字ベースの生チャンク（1000〜2000 token）」とメタデータを生成し、ベクトルDBに登録する。
3. **候補抽出（広域検索）**: ユーザーの質問に対し、まずはベクトルDBからテキスト検索で広範に生チャンクを抽出する（`top_k=50〜100`）。
4. **フルソースの自動フェッチ（オートローディング）**: 
   - 抽出された生チャンクのメタデータ（`file_uri`）を参照し、システムが裏側で**元のSSOTファイル全体（画像や図表を含むオリジナルのPDFなど）をストレージから自動的に取得**する。
   - *【生チャンクとの根本的な差】*: 生チャンクが「文脈から切り離された断片（パズルのピース）」に過ぎないのに対し、フルソースは「目次構造、前後の文章レイアウト、ページをまたぐ参照関係」などを保持した「完全な文脈」です。大前提として、**「RAGシステムがデータベースから該当箇所を探し出す能力（検索性能）」と「AIが提供された文章を読んで回答を作る能力（LLMの生成性能）」は全くの別物**です。生チャンクのみを渡す従来の手法では、前段の「検索」が取りこぼした時点で、後段のAIがどんなに優秀でも正しい回答は絶対にできません。フルソースを丸ごと読み込ませることで、検索システムの限界（ポンコツさ）に依存せず、AI自身の圧倒的な読解力で直接情報を探し出せるようになり、チャンク分割によって生じる「意味の喪失」や「前提条件の読み落とし」といった構造的欠陥を根本から排除できます。
   - ※テキストのみで十分な軽量クエリの場合はこのステップをスキップし、生チャンクのみを渡す（動的切り替え）。
5. **プロンプト合成とキャッシュ化**: 取得したフルソース（または生チャンク群）をプロンプトの先頭に配置し、「プロンプトキャッシュ」の対象とする。続いてユーザーの入力やチャット履歴を合算しLLMに投入する。
6. **生成と引用（Citation）の強制**: マルチモーダルLLMがフルソース（画像やレイアウト含む）を解析し、回答を生成。末尾に「どのファイルの、どのページ・図表を参照したか」を厳密に引用させる。
7. **SSOT監査ログ**: 生成に使われたファイル履歴をログに残し、情報の鮮度や利用頻度を追跡する。

   - 合計コンテキスト: 400,000 tokens（約160万文字）

   - 説明: 本書では標準モデル想定を基準に設計例を示す。実運用では各項目を減らして上限内（200K〜280K）に収める。

**高性能モデル（入力500K / 出力500K 想定）**
   - 合計コンテキスト: 1,000,000 tokens（約400万文字）
