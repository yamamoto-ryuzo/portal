# 第3章 既存APIを活用したSSOT抽出フローの設計

## 3.1 技術要素（RAGコアレイヤー）

実装パイプラインで使用する主要技術要素を示す。

| 技術要素 | 役割 | 主な効果 |
|---|---|---|
| **GraphRAG** | 法令・基準間の関係をグラフ構造で保持 | 多段推論（2ホップ以上）の実現 |
| **VectorRAG** | 非構造化テキストの意味検索 | 類似事例・実務知識の高速参照 |
| **ドメインオントロジー** | 概念・用語の公式スキーマ定義 | 用語の一意化・推論ルールの基盤 |
| **CogGRAG** | 人間の思考プロセスを模倣した問題分解 | 複雑な問いへの構造化した推論 |

パイプライン配置順: **入力整形 → VectorRAG → GraphRAG → CogGRAG（複雑な問いのみ） → 統合・生成**

### 3.1.1 MDファイル・システムプロンプトによる簡易代替

フル実装が困難な初期段階では、各技術をMDファイルとシステムプロンプトで代替できる。対応範囲と限界を下表に示す。

| 技術（§3） | 簡易実装での代替 | 実現できること | 主な限界 |
|---|---|---|---|
| **GraphRAG** | `knowledge_map.md`（Mermaid図）| 3ホップ程度の関係追跡・事実抽出 | 動的グラフ走査・大規模ノード管理不可 |
| **VectorRAG** | `source_docs/`（MD群を直接RAGへ登録） | 意味的類似検索・メタデータフィルタ | Cross-Encoder再ランキング・数万件超は非対応 |
| **CogGRAG** | `system_prompt.md`（回答手順として明示） | 固定パターンの多段推論・手順強制 | 動的分解・並列検索・自動矛盾検出は不可 |
| **ドメインオントロジー** | `entity_dictionary.md`＋制約ルール | 用語正規化・明示した制約の遵守 | SHACL網羅検証・OWL自動推論は不可 |

> 簡易代替は Track A（PoC段階）での早期検証に有効。精度・スケールの限界に達した時点でフル実装（4章）へ移行する。


## 3.2 RAGパイプライン機能を持つシステム一覧

**RAGパイプラインはAIのファインチューニングとは無関係であり、本質は検索システムの構築である。**

> **RAGとファインチューニングの根本的な違い**
>
> | 観点 | ファインチューニング | RAGパイプライン |
> |---|---|---|
> | **何を変えるか** | LLMのモデルパラメータ（重み） | ベクトルDB（外部の検索インデックス） |
> | **LLMは変わるか** | 変わる（再学習） | **変わらない** |
> | **回答品質を決めるもの** | モデルの能力 | **検索の精度**（チャンク・埋め込み・クエリ設計） |
> | **改善の作業** | GPU学習・データ整備 | **文書整備・インデックス設計・検索チューニング** |
>
> RAGパイプラインをいくら改善しても、LLMは賢くならない。  
> 逆に、どれだけ優れたLLMを使っても、インデックスが粗ければ組織固有の知識は取り出せない。  
> **RAGの品質 = 検索システムの品質** である。

> **RAGの品質を決めるのはフレームワークではなく検索システムの選択である**  
> AutoGen・LangGraph・CrewAI といったフレームワークは「エージェントの指示・連携・ツール呼び出し」を担うオーケストレーション層であり、  
> RAGの検索品質には直接関与しない。  
>  
> RAGで最も重要なのは「クエリに対して適切な文書がヒットすること」であり、  
> これを決めるのは以下の検索システム側の設計である：  
>  
> | 検索品質を左右する要素 | 具体的な選択・設計内容 |
> |---|---|
> | **インデックスシステム** | Azure AI Search / Chroma / Qdrant / OpenSearch 等の選択 |
> | **埋め込みモデル** | text-embedding-3-small / ada-002 / BGE 等の選択とドメイン適合性 |
> | **チャンク戦略** | チャンクサイズ・オーバーラップ・分割単位（段落/文/見出し） |
> | **検索戦略** | ベクトル検索 / ハイブリッド検索 / セマンティックランキング |
> | **再ランキング** | Cross-Encoder による検索結果の再順位付け |
>  
> フレームワークはこの検索システムを「ツール」として呼び出すだけである。  
> **AutoGen + Azure AI Search** も **CrewAI + Chroma** も、フレームワークが異なっても  
> 検索システムの設計次第でRAG品質は大きく変わる。  
>  
> 本書が §1.7 の表で評価しているのも「インデックス化が自動化されているか」という検索システム観点であり、  
> フレームワーク選定（§1.4）とは独立した評価軸である。

| システム | インデックス化の自動化 | 取得件数（AIへの渡し） | ベクトル化処理の実体 | 備考 |
|---|:---:|---|---|---|
| **LlamaIndex** | ◎ | 設定可（`similarity_top_k`、デフォルト2〜4件） | ドキュメントを指定するとチャンク・埋め込み・ベクトルストア格納まで自動実行。RAGインデックス化が中核機能 | インデックス化の最も信頼性が高い |
| **Dify（Knowledge Base）** | ◎ | 設定可（GUIで変更、デフォルト3件・最大20件） | ファイルをアップロードするとチャンク分割・埋め込み・インデックス化を自動実行。チャンク設定もGUIで変更可 | Track A（PoC）の主力 |
| **Azure AI Foundry（AI Search）** | ◎ | 設定可（デフォルト3〜5件） | Azure AI Search がドキュメントをチャンク・ベクトル化してインデックス。インデックス状態をポータルで確認可能 | インデックス品質の可視化が最も充実 |
| **Amazon Bedrock Knowledge Bases** | ◎ | 設定可（デフォルト5件・最大100件） | S3に置いたドキュメントをBedrock KBがチャンク・埋め込みし OpenSearch Serverless に格納 | AWS完結で管理が容易 |
| **Google Vertex AI RAG Engine** | ◎ | 設定可（デフォルト10件） | Google Cloud Storageのドキュメントを自動チャンク・埋め込みし Vertex AI Vector Search に格納 | GraphRAG対応も提供 |
| **AnythingLLM** | ○ | 設定可（デフォルト4件） | アップロードした文書をChroma等にベクトル化して格納。ファイル単位でインデックス状態を確認できる | セルフホスト可・UIが直感的 |
| **OpenWebUI** | ○ | 設定可（デフォルト5件） | ドキュメントをChroma等にベクトル化して格納。コレクション単位で管理 | セルフホスト可 |
| **CrewAI（Knowledge class）** | ○ | 設定可（コードで指定） | `MDXKnowledgeSource` 等でファイルを指定するとフレームワーク内蔵のベクトルストアにインデックス化 | コード1行で指定可能 |
| **Flowise** | ○ | 設定可（ノード設定で変更可） | VectorStoreノードにドキュメントローダーを接続するとインデックス化。ノード設定でチャンク・埋め込みを選択可 | GUI構築 |
| **AutoGen / LangGraph** | △ | 外部ストア依存・設定可 | **フレームワーク自身はインデックス機能を持たない**。Azure AI Search・Chroma・Qdrant等の外部ベクトルストアを別途構築し、ツールとして接続する必要がある | 外部ストアと組み合わせれば◎ |
| **LangChain / Semantic Kernel** | △ | 設定可（コードで指定） | 同上。外部ベクトルストアへのDocLoader・Embedderを自分で実装して初めてインデックス化が起きる | |
| **Microsoft Graph API** | △ | **非制御**（約10〜20件・Microsoftが決定） | SharePoint/OneDriveの**既存のMicrosoftインデックス**を利用。新規ドキュメントのインデックス化タイミング・チャンク戦略はユーザーが制御できない | RAGではなくMicrosoft Graph検索（M365 Copilotも同様） |
| **Box API** | △ | **非制御**（Box Search結果に依存） | Boxの**既存の内部インデックス**を利用。ユーザーがベクトル化・チャンク設計を操作できない | Box内コンテンツに限定 |

> **凡例** ◎ = 指定と同時に自動でインデックス化が行われる  ○ = 設定すればインデックス化される  △ = 外部依存または不透明

> **AutoGen / LangGraph の補足**  
> これらはフレームワークであり「どのベクトルストアを使うか」を選択できる設計になっている。  
> Azure AI Search（◎）を外部ストアとして接続すれば、LlamaIndex相当のインデックス品質を実現できる。  
> 「AutoGenを使う＝RAGのインデックス化は別途設計が必要」であることを認識しておく必要がある。

> **Microsoft Graph API / Box API の補足**  
> どちらも「既存のクラウドインデックス（Microsoft Graph / Box Search）を参照している」のが実態であり、  
> ユーザーが文書を追加してもインデックス反映のタイミング・品質は制御できない。  
> 本書では両者を「RAG知識を指定するシステム」ではなく「検索APIを持つ外部サービス」として扱い、  
> AutoGen のツールとして MCP Server 経由で接続する構成を採る。

### 3.2.1 選定の考え方

```
目的・制約
 │
 ├─ すぐ動かしたい（PoC）・インデックス化を手動で意識したくない
 │    └─ Dify → Track A（2章）
 │
 ├─ 本番品質・マルチエージェント・インデックス設計を自分で制御
 │    └─ AutoGen / LangGraph + Azure AI Search → Track B（3〜9章）
 │
 ├─ クラウド完結・インデックス管理もクラウドに任せる

```

> Microsoft Graph API・Box API はインデックス設計が不可なため、Track B の出力先・検索ツールとして接続する位置づけとなる。詳細は 3章・8章を参照。



## 3.3 なぜ今、高度な検索フローが必要なのか
第1章・第2章で見てきた通り、近年のLLMは数十万トークン規模の広大なコンテキストウィンドウを持っています。しかし、「Garbage In, Garbage Out（ゴミを入れればゴミが出てくる）」の原則は変わりません。大容量コンテキスト時代において最も重要になるのは、**「RAGの巨大なデータベースの中から、SSOT（唯一の正しい情報源）だけを、ノイズを混ぜずに安定して抽出する技術」**です。

## 3.4 実装の現実解：既存APIを「最高の資産」とする
SSOTを抽出するために、強力なハイブリッド検索や権限フィルターを一から構築（スクラッチ開発）する必要はありません。
実際のビジネス現場においては、すでに稼働しているドキュメント管理システムやエンタープライズ検索エンジン（Azure AI Search, Elasticsearchなど）の**「検索API」を活用する**のが最も現実的で効率的なアプローチです。

**このAPIは、まさにRAG全体を高速に検索できる「最高の資産」です。**
泥臭い検索機能は既存システムに任せ、開発側は「抽出された情報をどう処理し、SSOTとしてLLMに渡すか」という**オーケストレーション**にリソースを集中させます。

## 3.5 最高の資産からSSOTを抽出する高度な技術要素
この検索APIから得られた情報群の中から、複雑な文脈を持つ真のSSOTを確実かつ高度に抽出・推論するために、以下の次世代技術要素を組み合わせます。

| 技術要素 | 役割 | 主な効果 |
|---|---|---|
| **GraphRAG** | 法令・基準間の関係をグラフ構造で保持 | 多段推論（2ホップ以上）の実現 |
| **VectorRAG** | 非構造化テキストの意味検索 | 類似事例・実務知識の高速参照 |
| **ドメインオントロジー** | 概念・用語の公式スキーマ定義 | 用語の一意化・推論ルールの基盤 |
| **CogGRAG** | 人間の思考プロセスを模倣した問題分解 | 複雑な問いへの構造化した推論 |

## 3.6 SSOT抽出の高度化フロー（API連携の4ステップ）
上記の技術要素を実際のシステムフローに落とし込み、既存の検索APIと連携してSSOTを段階的に精製・抽出するプロセスは以下の順序で行われます。

### 3.6.1 意図分解と用語の正規化（CogGRAG × ドメインオントロジー）
- ユーザーの曖昧で複雑な質問を、**CogGRAG**の仕組みを用いて複数の小さな検索可能な問い（サブクエリ）に分解します。
- 同時に、**ドメインオントロジー**を参照し、ユーザー独自の言い回しや揺れのある用語を、APIが理解できる「公式な用語（スキーマ）」に変換（一意化）します。

### 3.6.2 広域検索と類似事例の取得（検索API × VectorRAG）
- 分解・正規化されたクエリを用いて、最高の資産である**既存の検索API（Box APIなどのエンタープライズ検索基盤、およびVectorRAG相当の機能を含む）**を叩きます。
- ここでは、ハイブリッド検索（意味検索×キーワード検索）やメタデータによる厳密なフィルタリング（最新版のみ、承認済みのみ、SSOTタグ付きのみ等）をAPI側で処理させ、関連しそうな実務知識や過去の類似事例を高速かつ広く取得（Recall）します。特にBoxのようなメタデータ管理に優れたクラウドストレージをSSOT基盤とすることで、AIは迷うことなく正確な正本（マスターデータ）にアクセスできます。

### 3.6.3 関係性の探索と文脈の補完（GraphRAG）
- 検索APIから得られた断片的な結果に対し、**GraphRAG**のナレッジグラフを適用します。
- 「Aという基準があるなら、関連するBという例外規定も引く必要がある」といった、単純な検索では見落とされがちな「多段推論（2ホップ以上のリンク）」を行い、文脈上欠かせない周辺のSSOTを補完します。

### 3.6.4 最終選定（リランキング）とLLMへの注入
- 集められ、関係づけられた情報群に対し、より高度な推論モデル（Cross-Encoderなど）を用いて**リランキング（再スコアリング）**を行います。
- ノイズを下位に押しやり、純度の高い真のSSOTセットだけをプロンプトの最上位に配置して、大容量コンテキストLLMへと渡します。

## 3.7 まとめ：検索設計がRAGの品質を決める
どれほど高性能なLLMを採用したとしても、**「抽出されたデータがSSOTでなければ、回答は信頼できない」**という事実は揺るぎません。
「とりあえず検索して上位をLLMに渡す」という初期のRAG構築から脱却し、既存の検索APIを基盤として、意図分解、関係性推論、リランキングを統合的に設計すること。これこそが、SSOTを安定して抽出し、業務で真に使い物になるRAGシステムを構築するための現在最も必要なアプローチです。

## 3.8 RAGの限界と次なる展開

ここまで、GraphRAG、VectorRAG、ドメインオントロジー、CogGRAGという4つの高度なRAG要素技術を駆使した検索・抽出フローを解説しました。しかし、RAGという仕組みそのものには、アーキテクチャ上の根本的な限界が存在します。

それは、**「RAGは所詮、質問された瞬間に外部からカンペ（コンテキスト）を持ってくるだけの仕組みに過ぎない」**ということです。AIモデル自身がそのドメインの知識を根本的に理解し、血肉としているわけではありません。
人間の専門家に例えるなら、RAGは「質問されてからマニュアルを高速で検索して読み上げる素人」です。真の専門家のように、言われずとも背後にある文脈や業界の常識を**「背景知識」**として踏まえて推論するには、AIモデルの内部構造そのものに知識を定着させる必要があります。汎用AIモデルの進化（一般的な賢さの向上）とは異なる次元で、独自のドメイン知識をモデルの「背景」として組み込む手法への転換が求められています。

このパラダイムシフトを実現するための代表的なアプローチとその周辺技術は以下の通りです。

### 3.8.1 LoRA (Low-Rank Adaptation) / PEFTによる知識の焼き付け
RAGの次なる展開として最も注目されるのが、**PEFT（Parameter-Efficient Fine-Tuning）**、とりわけ**LoRA**を活用したアプローチです。
LoRAは、巨大な基盤モデルの元の重み（汎用的な知能）を凍結したまま、低ランクの行列という極めて軽量な追加モジュールのみを学習させます。これにより、汎用AIの進化の恩恵を受けつつ、自社特有の専門用語、文体、暗黙知といった「背景」をモデルに焼き付けることができます。推論時に検索のオーバーヘッドが発生せず、無意識レベルで専門家として振る舞うことが可能になります。

### 3.8.2 RAFT (Retrieval Augmented Fine Tuning)
RAGとFine-tuningのハイブリッドである**RAFT**も有力な手法です。
これは単に知識をモデルに記憶させるのではなく、「RAGが検索してきたドメイン特有の複雑な文書を、どう正しく読み解き、ノイズを無視して回答を導き出すか」という**推論プロセスそのものをファインチューニング**します。RAGの動的な情報更新能力を活かしつつ、読解力という背景能力を強化するアプローチです。

### 3.8.3 Continuous Pre-training (継続的事前学習)
もしLoRAのような表面的な適応では不十分な場合、さらに深く根本的な言語理解を図る手法として継続的事前学習があります。社内の膨大なドキュメントや業界特有のテキストを用いて、事前学習のフェーズを独自に延長させます。コストと計算資源は跳ね上がりますが、独自のオントロジーや概念体系をAIのニューロンの最深部に刻み込むことができます。

### 3.8.4 擬似的な背景化：Prompt Caching (コンテキストキャッシュ)
モデルのパラメータを直接更新しない代替アプローチとして、最近台頭しているのが**Prompt Caching**です。
数十万トークンにおよぶ社内の基本規程やSSOTの塊を、システムプロンプトとして常にAPI側にキャッシュ（常駐）させておきます。ファインチューニングのコストをかけずに、モデルが常に広大な背景知識を前提として対話できる環境を擬似的に作り出します。

## 3.9 RAG資産を活用した「背景知識」学習への移行戦略

単にテキストドキュメントをそのまま読み込ませてLoRA学習等を行うだけでは、AIは「知識の複雑な関係性」を正しく把握できません。そこで、これまで構築した**「4つの高度なRAG技術」のデータ構造とプロセスそのものを、ファインチューニングの高品質な学習データセットとして再利用する**という強力なアプローチが存在します。

RAGは抽出の仕組みであると同時に、**AIに背景知識の「関係性と思考法」を教え込むための「最高の教材生成エンジン」**として機能します。

### 3.9.1 GraphRAGの「関係性」を推論データに変換（関係性の背景化）
GraphRAGが構築したナレッジグラフ（ノードとエッジのネットワーク）は、そのままではモデルの重みとして学習できません。しかし、「Aという法面工法は、Bという地質条件で適用される（制約事項：C）」といったグラフ上の構造を、LLMを用いてQ&A形式の自然言語テキストに自動変換（Synthetic Data Generation）します。この「関係性」を明記したデータでLoRA学習を行うことで、複雑な論理構造がモデルの内部ネットワークに直接焼き付けられます。

### 3.9.2 ドメインオントロジーを「基礎語彙」として学習（用語の背景化）
構築したドメインオントロジー（専門用語の厳密な定義や階層構造）は、そのまま継続的事前学習（Continuous Pre-training）やインストラクションチューニングのベースデータとして投入します。用語の揺れ、同義語、上位・下位概念の関係性がモデルの「基礎語彙」として定着するため、推論時にわざわざ辞書を引かなくても自然に正しい用語で思考できるようになります。

### 3.9.3 CogGRAGの「思考プロセス」をChain-of-Thoughtデータへ転用（推論の背景化）
CogGRAGが行う「ユーザーの曖昧な問いを、どのように複数のサブクエリに分解し、順序立てて結論を導き出すか」という実行ログは、非常に価値の高い**Chain-of-Thought（CoT：思考の連鎖）の教師データ**になります。この推論ステップの履歴をRAFTやLoRAで学習させることで、モデル自体が「プロの専門家のように問題を論理的に分解して考える」という思考の型を獲得します。

### 3.9.4 VectorRAGの「検索・選択ログ」を用いた嗜好学習（価値観の背景化）
実際の運用でVectorRAGが検索し、人間（あるいはリランカーモデル）が最終的に「正解のSSOT」として選んだドキュメントと、「不正解（ノイズ）」として弾かれたドキュメントのログを活用します。「この質問の文脈では、こちらの情報が適している」という選別のペアデータを蓄積し、DPO (Direct Preference Optimization) などの手法でモデルの好みを調整します。これにより、モデルは「どの情報をより信頼・重要視すべきか」というドメイン特有の価値観を身につけます。

### 3.9.5 次のステップへ：LoRAによる「個の強化」から「マルチエージェント」の組織化へ
4つのRAG技術による「完璧な情報の抽出」は、システム構築の重要なマイルストーンであると同時に、モデル自身を賢くするための**「高品質な学習データパイプライン」**の完成を意味します。RAGで得られたデータ資産をLoRA等に流し込むことで、外部検索に過度に依存しない「背景知識を持った真の専門家AI」を作り出すことができます。

ここで、「LoRAというガジェットを無制限に追加・同時に付け替えできれば、それが実質的にマルチエージェントになるのではないか？」という疑問が生じるかもしれません。確かに、1つの中規模LLMに対して「法務用LoRA」「技術用LoRA」といった複数のガジェットを同時に展開・付け替える（Multi-LoRAアプローチ）ことで、単一のベースAIからあらゆる分野の専門家チームを立ち上げることは可能です。（※なぜ単一の巨大モデルではなく、中規模モデル＋複数LoRAの同時展開がマルチエージェント構築において最適解なのかについては、ハードウェアリソースと専門性の混線回避の観点から第6章 6.10.4で詳述します）

しかし、**「複数のLoRAを付け替えること」と「マルチエージェント」は本質的に異なる概念です。**

*   **LoRAの切り替え（ガジェットの追加）：**
    1人の人間が、その都度「異なる専門ツール（ガジェット）」を持ち替えて**1人で全作業を行う**アプローチです。知識は豊富になりますが、結局は1つの脳（モデル）で考えているため、「自分の思考の偏りや思い込み（ハルシネーション）」に自分で気づくことができません。
*   **マルチエージェント（組織化）：**
    複数の独立したAIが**「チーム」として分業・対話・監視を行う**アプローチです。例えば、LoRAで強化された「作成者AI」が書いたドラフトを、あえて別の役割を与えられた「査読者AI」が批判的にチェックし、「進行管理AI」がタスクの進捗を管理する、といった**自律的な相互作用**が発生します。

すなわち、LoRA（ガジェットの無制限な追加）が「1人の天才を育成し、無数のスキルを付与すること」であるならば、次なる課題であるマルチエージェントとは「その天才たちや、ツールを操るスタッフたちを集めて、互いに議論・検証させながら業務を完遂する『組織』を作ること」です。

次章では、この強化された個々のモデル（LoRA）や情報検索ツール（RAG）をどのように組み合わせ、互いにチェックし合いながら自律的にタスクを進めるシステムへと統合するかという**「マルチエージェント構成」**のアーキテクチャ設計と実装フェーズへと進みます。
