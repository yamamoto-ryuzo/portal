# 3. 採用技術の詳細

### 3.1 GraphRAG（ナレッジグラフ検索）

GraphRAGは法令・技術基準・マニュアル間の構造的な関係を知識グラフとして保持し、多段の依存関係を正確にトレースする検索基盤である。

#### GraphRAGが果たす役割

- **構造的関係の保持**: 法令・技術基準・マニュアル間の「依存関係」「包含関係」「委任関係」をノード（実体）とエッジ（関係）として表現する。
- **明示的事実の抽出**: 文書に直接記載された事実（条文番号・適用条件・参照先）を正確に取得する。
- **推論パスの可視化**: 「なぜその回答に至ったか」という根拠パスを応答とともに提示できる。

#### 適した問い

「〇〇工法の適用を制限する河川法の条文はどれか」など、事実に基づき明確に特定できる**抽出的な問い**。複数文書をまたぐ参照関係の追跡に特に有効。

#### グラフの構造（ノード・エッジ種別）

| 種別 | 例 |
|---|---|
| ノード（法令） | 河川法第26条、道路法第32条 |
| ノード（技術基準） | 河川砂防技術基準（調査編） |
| ノード（工法） | 堤防掘削工法、盛土補強工法 |
| エッジ（適用する） | 河川法第26条 → 工作物新築許可 |
| エッジ（参照する） | 技術基準 → 根拠法令 |
| エッジ（委任する） | 法律 → 施行令 → 告示 |

#### Cypherクエリサンプル（説明用）

```cypher
MATCH (h:工法 {name: "堤防掘削"})-[:準拠すべき]->(s:技術基準)
      -[:参照する]->(l:法令)
RETURN l.name, l.条文番号, s.name
```

#### システム統合ポイント

1. **文書取込**: 法令・基準文書からトリプル（主語・述語・目的語）を抽出しグラフDBへ投入する。
2. **エンティティリンク**: ドメインオントロジー（3.4）のエントリと照合し、ノードIDを統一する。
3. **クエリ実行**: SPARQL／GremlinでオントロジースキーマにもとづいたGraph探索を行う。
4. **CogGRAGとの連携**: 各サブプロブレム（3.3）の「グローバル検索」フェーズでGraphRAGを呼び出す。
5. **回答生成**: 取得したパス情報を根拠ノードリストとしてLLMのプロンプトに付与する。

### 3.2 VectorRAG（ベクトル類似度検索）

VectorRAGは非構造化テキストを高次元ベクトルに変換し、意味的な類似性にもとづいて関連事例を横断的に検索する。GraphRAGが「構造的な事実」を扱うのに対し、VectorRAGは「文書に潜む暗黙の傾向・文脈」を引き出す役割を担う。

#### VectorRAGが果たす役割

- **非構造化データの活用**: 施工トラブル報告書・現場進捗記録・議事録などの自由記述テキストを検索対象とする。
- **意味的類似検索**: 表記の揺れや言い回しの違いを超え、意味的に近い文書チャンクを取得する。
- **事例横断的なパターン抽出**: 複数現場の報告書から共通するリスク傾向を浮かび上がらせる。

#### 適した問い

「過去の軟弱地盤対策におけるトラブルの共通傾向は何か」など、生データに直接明示されていない抽象的なニュアンスを汲み取る**要約的な問い**。類似事例の参照によるリスク予測にも有効。

#### ベクトル検索の仕組み

| ステップ | 内容 |
|---|---|
| チャンク分割 | 文書を意味のある単位（段落・セクション）に分割する |
| エンベディング生成 | 各チャンクをLLM埋め込みモデルで高次元ベクトルに変換する |
| インデックス構築 | FAISSやpgvectorなどのベクトルDBにインデックスを格納する |
| クエリ変換 | 入力クエリも同一モデルでベクトル化する |
| 類似度計算 | コサイン類似度等でTop-Kチャンクを取得する |
| 再ランキング | Cross-Encoderでチャンクの関連度をスコアリングし精度を向上させる |

#### ベクトル検索サンプル（説明用）

```python
# クエリのベクトル化と類似チャンク取得のイメージ
query = "軟弱地盤における盛土施工時のトラブル事例"

# 1. クエリをベクトル化
query_vector = embedding_model.encode(query)

# 2. ベクトルDBからTop-5チャンクを取得（コサイン類似度）
results = vector_db.similarity_search(
    vector=query_vector,
    top_k=5,
    filter={"工事区分": "河川工事"}   # メタデータでフィルタ
)

# 3. 結果例
# [0.92] 施工報告書2022-X号 p.14「盛土完了後に地盤沈下が発生…」
# [0.89] 施工報告書2019-Y号 p.8 「軟弱層確認後の対策遅延により…」
# [0.85] 現場日誌2023-Z工区  p.3 「出水期前の地盤改良が未完了…」
```

#### システム統合ポイント

1. **文書取込**: 報告書・記録をPDFパーサで抽出し、チャンク分割後にエンベディングを生成する。
2. **メタデータ付与**: 工事番号・発注者・日付などをペイロードとして各ベクトルに付与し、フィルタ検索を可能にする。
3. **ハイブリッド検索**: BM25（キーワード検索）とベクトル検索のスコアをRRF（Reciprocal Rank Fusion）で統合し再現率を高める。
4. **CogGRAGとの連携**: 各サブプロブレム（3.3）の「ローカル検索」フェーズでVectorRAGを呼び出す。
5. **回答生成**: 取得チャンクをソース情報（文書名・ページ）とともにLLMのコンテキストに付与する。

### 3.3 CogGRAG（認知思考型RAG）

CogGRAGは人間の専門家が複雑な問題を解くときの「思考の構造化」をAIで再現するオーケストレーションフレームワークである。GraphRAG（3.1）とVectorRAG（3.2）の両検索エンジンを指揮し、段階的推論によって複合的な問いに対応する。

#### CogGRAGが果たす役割

- **思考の分解と統合**: 複雑な問いをマインドマップ状の「木構造」に自動分解し、サブプロブレムごとに最適な検索手段を割り当て、ボトムアップで結論を統合する。
- **カスケードエラーの防止**: 一足飛びの推論を排し、各ステップの根拠を積み上げることで誤答の連鎖を防ぐ。
- **検索戦略の制御**: GraphRAG（構造的事実）とVectorRAG（意味的類似）を問いの性質に応じて使い分け・組み合わせる。

#### 適した問い

「この工区の設計変更は河川法・環境影響評価法・労働安全衛生法のどの手続きに影響するか」など、複数の法令・技術基準・現場状況を横断して判断が必要な**複合的・多段的な問い**。

#### 思考分解のステップ

| ステップ | 内容 |
|---|---|
| ① 問いの解析 | 入力クエリをLLMで意図分類し、必要な知識ドメインを特定する |
| ② 木構造生成 | 問いを「サブプロブレムの木」に分解する（例：法令確認・技術基準確認・リスク評価） |
| ③ リーフ検索 | 末端ノード（具体的な問い）にGraphRAGまたはVectorRAGを割り当てて検索する |
| ④ ボトムアップ統合 | 各リーフの回答を親ノードへ順次集約し、矛盾・欠落を検出する |
| ⑤ 最終回答生成 | ルートノードでオントロジー制約（3.4）と整合性を確認の上、LLMが回答を生成する |

#### 思考木サンプル（説明用）

```
問い: 「堤防掘削工事の設計変更に必要な手続きは？」
├── [GraphRAG] 河川法上の許可申請要件は？
│   └── → 河川法第26条：工作物新築・改築許可
├── [GraphRAG] 土木工事標準仕様書の変更届出基準は？
│   └── → 設計変更ガイドライン第3章
└── [VectorRAG] 過去の類似変更案件における留意点は？
    └── → 施工報告書2022-X号（地盤沈下リスク）
統合: 上記3点を根拠に手続きフローを回答
```

#### システム統合ポイント

1. **オーケストレーター**: LangGraph／AutoGenなどのエージェントフレームワークで木構造の実行フローを管理する。
2. **検索ルーター**: サブプロブレムの種別（事実抽出／類似検索）に応じてGraphRAG・VectorRAGへルーティングする。
3. **中間結果キャッシュ**: 再利用可能なサブプロブレム回答をセッションキャッシュに保存し、応答速度を向上させる。
4. **オントロジー制約注入**: 各思考ステップにオントロジー由来の制約ルール（3.4）を付与し、LLMへ渡す。
5. **根拠トレース**: 最終回答に用いた全ノード・文書ソースをトレースログとして保持し、監査に対応する。

### 3.4 ドメインオントロジー（知識の公式スキーマ）

オントロジーはシステム全体の「公式用語辞書」かつ「推論の基盤」であり、GraphRAGとCogGRAGの品質を根底から支える。

#### オントロジーが果たす役割

- **用語の一意定義**: 同義語・略語を正規化し、同一エンティティとして一貫して扱う（例: 「設計洪水位」「H.W.L」「計画高水位」を同一ノードに統合）。
- **概念間関係の明示**: 包含・依存・適用条件などの関係タイプを型として定義し、推論の精度を高める。
- **推論ルールの基盤**: 「条文Aが適用される場合、手続きXが必要」というルールをSHACLやOWL公理として表現し、自動判定を可能にする。

#### 土木事業管理オントロジーの最小スキーマ

| クラス | 主要プロパティ | 具体例 |
|---|---|---|
| `法令` | 施行年、所管省庁、委任先 | 河川法、道路法 |
| `技術基準` | 参照法令、工事区分、発行年度 | 河川砂防技術基準 |
| `工法` | 適用条件、制限条件、準拠基準 | 盛土補強工法 |
| `構造物` | 種別、適用法令、設計基準 | 堤防、橋梁 |
| `現場報告` | 工事番号、発注者、日付 | 施工報告書2024-X号 |
| `リスク要因` | 影響を与える工法・構造物 | 軟弱地盤、出水期 |

プロパティ（関係）: `適用する`, `参照する`, `定義する`, `委任する`, `影響を与える`, `準拠すべき`

#### TTLサンプル（説明用）

```turtle
@prefix ex: <http://example.org/ontology#> .
ex:河川法_第26条 a ex:法令 ;
    ex:適用する ex:工作物新築許可 .
ex:工作物新築許可 ex:適用される ex:堤防掘削 .
ex:堤防掘削 a ex:工法 ;
    ex:準拠すべき ex:河川砂防技術基準 .
```

#### オントロジーのシステム統合ポイント

1. **エンティティリンク**: 文書トークンをオントロジーのエントリにマップし、ノードIDを付与する。
2. **トリプル格納**: 抽出したトリプルをKnowledge Graph（RDF/Property Graph）へ投入し、ソース（文書ID/ページ）をノード・エッジプロパティとして保持する。
3. **制約検索**: GraphRAGはオントロジースキーマに基づくSPARQL/Gremlinクエリを実行する。
4. **推論エンジン**: OWL推論・SHACL検証・ルールエンジン（Drools等）でドメインルールと整合性検査を行う。
5. **生成フェーズ**: CogGRAGの各思考ステップにオントロジー由来の制約と根拠ノードリストを付与してLLMに渡す。
