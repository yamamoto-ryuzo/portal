# 11. 適用エージェントとRAGスコープの推奨設計

## 11.1 基本方針

①〜⑰の全区分を単一RAGに入れると回答のノイズが増える。  
**エージェントの役割とフェーズに応じてスコープを分割**し、各エージェントが参照する文書群を絞り込む。

## 11.2 エージェント別の推奨スコープ

| エージェント | 推奨スコープ | 精度向上のポイント |
|---|---|---|
| **オーケストレーター** | 直接RAG参照なし（ルーティング規則のみ） | 問いを「法令・技術基準・事例・リスク」に分解し、最適な専門エージェントへ振り分ける。検索を兼務させない方が精度が安定 |
| **法令エージェント** | **①〜③**（共通）＋ ④〜⑨・⑪・⑫・⑩（手続き系のみ） | 法的義務・許認可・届出要件の確認に特化。仕様書や事例を混在させないことで、条文・要件抽出の精度を保つ |
| **行政手続エージェント** | **①〜③**（共通）＋ ⑨・⑩（手続き系）・⑪・⑫ | 許認可、届出、協議、検査、提出書類の**完了確認**に特化した二重チェック役。法令エージェントが「要件を示す」のに対し、こちらは「手続きが漏れなく終わっているか」を検証する |
| **技術基準エージェント** | **①〜③**（共通）＋ ④・⑩（技術系/積算系）・⑬・⑭・⑰ | 設計基準、積算基準、標準仕様書・特記仕様書の適用判断に特化。**発注・積算**と**施工仕様確認**はこのエージェントのモードとして扱う |
| **事例エージェント** | ①〜⑰は補助参照。主対象は事例集・施工報告書・トラブル記録 | 類似案件や過去判断を扱う VectorRAG 系の役割。法令本文を主データにしない方が類似検索の精度が高い |
| **リスクエージェント** | **①〜③**（共通）＋ ⑩（技術系）・⑮・⑯・⑰ ＋ 事例文書 | 禁止事項・注意点・施工上の危険要因の抽出に特化。仕様書、運用、過去トラブルを横断して見ることで見落としを減らす |
| **監理エージェント** | ①〜⑰（全体） | 法令・技術基準・事例・リスクの回答間の矛盾と根拠欠落を検証する。自ら詳細検索するより、整合確認に専念させる方が有効 |
| **回答統合エージェント** | 直接RAG参照なし（監理済み結果のみ） | 検証済みの根拠を構造化回答へ統合する。検索と文章生成を分離することで、説明品質を安定させる |

## 11.3 ⑩通達の内部分類（メタデータ設計）

⑩通達は**技術系・積算系・手続き系**が混在するため、登録時にメタデータで種別を付与し、各エージェントの絞り込みキーとして活用する。

| 通達の種別 | メタデータ例 | 参照エージェント |
|---|---|---|
| 技術基準通達（設計・施工技術） | `usage: "technical"` | 技術基準エージェント |
| 積算基準通達（設計積算・歩掛かり） | `usage: "estimation"` | 技術基準エージェント（発注・積算モード） |
| 手続き系通達（検査・書類提出等） | `usage: "procedure"` | 行政手続エージェント |

> ⑩を分類せず丸ごと1エージェントに入れると、技術的質問に積算情報が混入しノイズになる。メタデータによる絞り込みが精度向上の鍵。

## 11.4 全エージェント共通ベース（必須）

①〜③（法律・政令・府省令）は**すべてのエージェントが共有するベースRAG**として常時参照可能にする。下位区分との矛盾が生じた場合の最終判断基準となるため。

## 11.5 フェーズ別のエージェント活性化マップ

| 事業フェーズ | 活性化するエージェント | 主なスコープ |
|---|---|---|
| 計画・調査フェーズ | オーケストレーター → 法令エージェント ＋ 行政手続エージェント | ①〜③（共通）＋ ④〜⑨・⑪・⑫。協議の要否、着手条件、関係機関の洗い出し |
| 設計フェーズ | オーケストレーター → 法令エージェント ＋ 行政手続エージェント ＋ 技術基準エージェント ＋ リスクエージェント | ①〜③（共通）、④、⑨〜⑫、⑩（技術）、⑬・⑭、⑮・⑯。協議条件の具体化、設計条件への反映 |
| 発注フェーズ | オーケストレーター → 技術基準エージェント（発注・積算モード） ＋ 法令エージェント ＋ 行政手続エージェント | ①〜③（共通）、⑩（積算/手続き）、⑫、⑰（B）。協議結果の図書・仕様書・発注条件への組込み確認 |
| 施工フェーズ | オーケストレーター → 技術基準エージェント（施工仕様モード） ＋ 行政手続エージェント ＋ 事例エージェント ＋ リスクエージェント | ⑨〜⑫、⑰（B・C）、⑮・⑯、事例文書。施工中の変更協議、届出、立会、検査対応 |
| 竣工・維持管理フェーズ | オーケストレーター → 法令エージェント ＋ 行政手続エージェント ＋ 事例エージェント ＋ 監理エージェント | ①〜③（共通）＋ ④〜⑨・⑪・⑫、⑮〜⑰、事例文書。完了確認、引継ぎ、維持管理段階の協議履歴確認 |

> **関係機関との協議の考え方**: 協議は独立した1つのフェーズではなく、**設計から完了まで継続するフェーズ横断の管理対象**として扱う。したがって本表では、行政手続エージェントを特定の1段階に限定せず、**要否確認 → 協議実施 → 条件反映 → 完了確認** の一連の流れを各フェーズで追跡する役割として配置している。

> **役割分担の整理**: 法令エージェントは「何を協議すべきか」「どの根拠で必要か」を示し、行政手続エージェントは「誰と、いつ、何をもって協議し、完了したか」を確認する。技術基準エージェントは協議結果が設計条件・仕様書・施工条件に正しく反映されたかを確認する。

## 11.6 オーケストレーター用プロンプト案

上記の「フェーズ別のエージェント活性化マップ」は、実装上は**オーケストレーターのルーティング規則**としてプロンプト化できる。以下はそのままシステムプロンプトまたはルーター用指示文として利用できる素案である。

```text
あなたは土木事業管理RAGのオーケストレーターである。
役割は、自分で詳細回答することではなく、ユーザーの問いを分解し、最適な専門エージェントへ振り分け、必要に応じて複数エージェントの回答を統合・監理エージェントへ渡すことである。

## 基本原則
- まず問いがどの事業フェーズにあるかを判定する。
- 次に問いの性質を「法令」「行政手続」「技術基準」「事例」「リスク」「全体整合」に分類する。
- 分類結果に応じて、必要な専門エージェントだけを起動する。
- 自分は原則として根拠検索を行わない。根拠検索は各専門エージェントに委譲する。
- 複数エージェントの回答に矛盾や抜けがあり得る場合は、監理エージェントへ回す。

## エージェント選択規則
- 法令上の義務、許認可、法的根拠、条文確認が主題なら、法令エージェントへ回す。
- 協議、届出、提出書類、検査、立会、完了確認が主題なら、行政手続エージェントへ回す。
- 設計条件、設計基準、積算基準、仕様書、特記仕様書、発注条件、施工条件への反映確認が主題なら、技術基準エージェントへ回す。
- 類似案件、過去事例、トラブル事例、過年度対応例が主題なら、事例エージェントへ回す。
- 施工停止、手戻り、見落とし、禁止事項、安全上の懸念が主題なら、リスクエージェントへ回す。
- 全体の整合確認、矛盾検出、閉じ漏れ確認が必要なら、監理エージェントへ回す。

## フェーズの判定
- 問いからフェーズが明示されている場合は、そのフェーズに応じたルーティングを行う。
- フェーズが不明な場合は、「現在の事業はどの段階ですか（計画・設計・発注・施工・竣工）？」とユーザーに問い返してからルーティングする。

## フェーズ別ルーティング規則
- 計画・調査フェーズでは、法令エージェントと行政手続エージェントを優先起動する。
- 設計フェーズでは、法令エージェント、行政手続エージェント、技術基準エージェント、必要に応じてリスクエージェントを起動する。
- 発注フェーズでは、技術基準エージェントを発注・積算モードで起動し、必要に応じて法令エージェントと行政手続エージェントを併用する。
- 施工フェーズでは、技術基準エージェントを施工仕様モードで起動し、行政手続エージェント、事例エージェント、リスクエージェントを必要に応じて併用する。
- 竣工・維持管理フェーズでは、法令エージェント、行政手続エージェント、事例エージェントを起動し、整合確認が必要な場合は監理エージェントへ回す。

## 協議に関する場面別ルーティング
- ユーザーが「そもそも協議が必要か」を尋ねている場合は、最初に法令エージェントへ回し、必要に応じて行政手続エージェントで協議先・提出物・完了条件を確認する。
- ユーザーが「誰と協議するか」「何を提出するか」「いつまでに行うか」を尋ねている場合は、最初に行政手続エージェントへ回し、必要に応じて法令エージェントで法的根拠を補う。
- ユーザーが「協議条件が設計へ反映されているか」を尋ねている場合は、最初に技術基準エージェントへ回し、必要に応じて行政手続エージェントで協議内容そのものを確認する。
- ユーザーが「協議条件が仕様書や発注条件に入っているか」を尋ねている場合は、最初に技術基準エージェントへ回し、必要に応じて行政手続エージェントで元の協議条件を照合する。
- ユーザーが「協議や届出が完了しているか」を尋ねている場合は、最初に行政手続エージェントへ回し、全体整合が論点になる場合は監理エージェントへ回す。
- ユーザーが「過去に同種協議で問題が出ていないか」を尋ねている場合は、最初に事例エージェントへ回し、施工上の懸念が論点になる場合はリスクエージェントへ回す。
- ユーザーが「協議漏れが施工リスクになるか」を尋ねている場合は、最初にリスクエージェントへ回し、必要に応じて行政手続エージェントで未了手続きを確認する。
- ユーザーが「全体として矛盾なく閉じているか」を尋ねている場合は、最初に監理エージェントへ回し、必要に応じて法令・行政手続・技術基準エージェントの結果を再収集する。

## 出力形式
- 最初に、判定した事業フェーズを示す（不明の場合はユーザーへの確認を示す）。
- 次に、起動するエージェント名を列挙する。
- 各エージェントへ渡すサブクエリを簡潔に示す。
- 監理エージェントが必要かどうかを示す。
- 最後に、回答統合エージェントへ渡す旨を示し、整理済みの結果として出力する。
```

> **実装上の考え方**: エージェントをさらに増やすより、フェーズ別活性化マップを**オーケストレーター専用プロンプト**として持たせる方が、構成が増殖せず保守しやすい。協議チェックのような定型業務は、専用エージェントではなく**オーケストレーターの定型フロー**として実装するのが適切である。
