
# 書類作成AI基本方針
書類作成AIの設計・運用における基本的な方針を以下に示します。

## 1. 編集プラットフォームと出力方針

### なぜ今これが必要か（Word-first前提）
現場の多くはWord（.docx）を中心に作業しており、Track Changesや複雑なレイアウトを前提とした運用ではMarkdown原本が現実的でない場合が多いです。現行ワークフローを壊さずにRAGや自動化を導入するには、`.docx`を公式原本とするアプローチが最も実用的です。

### 改善方針（要点）
- 原本管理：`.docx`を公式ソースとし、日常編集はWordで行う（Track Changes・コメントを標準運用）。
- メタデータ管理：ドキュメントプロパティ（カスタムプロパティ）もしくは同階層の `document.yaml` でテンプレ・責任者・ビルドフラグを定義。
 - 原本直接編集（MD生成は行わない）：編集は常に`.docx`で行い、中間のMarkdownファイルを生成・保存するワークフローは採用しません。CIやツールは原本から直接テキスト／メタを抽出してインデックス化・検索用埋め込みを作成できますが、抽出結果は一時的に扱い永続化しません。

 - 安定ID付与：章・節・ブロック単位でContent Controls（SDT）やカスタムXMLを利用し安定したIDを埋め込むことで、抽出時のチャンクと原文の部分を確実に対応付けます。

 - 差分・レビュー：差分確認はWordのTrack Changes/コメントと、CIで生成する高忠実度PDFで行い、編集は原本で完結します。Pull Requestベースのレビューフローが必要な場合はPDFを差分表示用アーティファクトとして用います。

 - 直接編集支援：必要に応じてWord Add-inやOffice Online連携で提案・承認ワークフローを構築し、編集提案を原本に反映する際は管理者承認を必須とします。

 - 変換の取り扱い：外部共有用の派生物（PDF等）は生成しますが、Markdownなどの変換成果物を編集ソースとせず、ラウンドトリップ編集は基本的に行いません。

 - RAG連携：埋め込み・チャンク化は原本からの一時抽出を用いて行い、各チャンクには必ず元ファイルのID・作成日・更新日を紐付けます。

 - ガバナンス：重要な書式や複雑レイアウトの変更はWord上で手動確認・承認を行い、自動反映は管理者承認制で段階的に拡張します。

この方式により、`.docx`原本の編集性を最優先に保ちながら、差分確認・インデックス化・RAG連携を実現します。まずは章本文（テキスト主体）を対象に運用を開始し、表や脚注は段階的にサポート範囲を広げてください。
- バージョン管理とストレージ：原本はGit LFS、SharePoint、OneDrive等のバイナリ対応ストレージで管理し、差分・レビューは自動生成Markdown/PDFで行う。
- Round‑tripポリシー：完全往復は保証せず、表や細かい書式は`.docx`側で保持。変換は主に読み出し（インデックス化／RAG用）に限定する。
- RAG連携：埋め込み・チャンク化は抽出テキストを用い、各チャンクに元ファイルのID・作成日・更新日・セクション参照を付与する。
- 出力と配布：最終配布は原本のPDF/`.docx`を基本とし、外部共有用にPandocで変換したフォーマットを併用する。

### 具体的運用ルール（短め）
1. 編集は常に原本`.docx`で行い、保存先は組織が採用するバイナリストレージに置く。
2. プルリクやレビューは自動生成したPDF（CI出力）を用いる。編集差分はWordのコメント/履歴で判断。
3. CIで抽出されたMarkdownはRAGインデックス作成・検索用にのみ用い、手動編集は不可とする。
4. 重要テンプレートは`.docx`テンプレ（.dotx）として管理し、テンプレのバージョンをドキュメントプロパティで参照する。

### 推奨MVP実装（最小限）
- Wordの原本をリポジトリで管理（Git LFSなど）→ Git push でCIトリガー
- CIジョブ：原本から直接テキスト／メタを抽出し（中間Markdownは生成しない）、抽出結果を一時的にベクトル化してRAGに登録（注：抽出結果は保存しない）
- 出力：CIで高忠実度のPDFを生成し、レビュー用アーティファクトとして添付（レビューでの編集は原本のWordで行う）

この章はWord-first運用を前提に、原本の忠実性を最優先にしつつCIでの読み出し（インデックス化・レビュー生成）を行う方針に修正しました。

## 2. RAG（知識拡張）方針
- 概要: ローカル／クラウド／内部API／外部公開を統一的に取り込み、RAGパイプライン（正規化→チャンク化→埋め込み→ベクトルDB→リトリーバル→コンテキスト注入）で活用する。
- ソース分類: ローカル（高優先）、クラウド（高〜中）、内部サービス/API（中）、外部公開（低〜中）。
- 同期と認可: ソースごとに認可方式と同期ポリシー（オンデマンド・定期・手動）を定義。OAuth等で最小権限を採用。
- 品質管理: チャンク品質、出典紐付け、PII検出・マスキング、更新履歴を必須とする。

## 3. データ形式と変換ポリシー
- 対応フォーマット: Word（.docx）、Markdown（.md）、PDF（OCR含む）、PowerPoint（.pptx）、Excel/CSV、JSON/API、音声、画像。
- 変換ポリシー: 入力→標準中間形式（UTF-8テキスト＋メタデータ）→チャンク化→埋め込みの流れを基本とする。
- チャンク設計: 512–2048トークン目安。表・脚注・注釈はメタデータとして保持。
- 特記事項: PDF/画像はOCR品質評価を行い、低品質時は手動確認を要求。

## 4. セキュリティ・ガバナンス
- アクセス制御: 機密ソースは暗号化ストレージに格納し、RBACでアクセスを制御。
- 監査ログ: 入力・検索・出力・承認の全ログを保持し、監査可能にする。
- 承認フロー: 重要文書は人間承認を必須化し、出典と信頼度を提示。
- PII対策: 埋め込み前にPII検出・マスキング、監査記録を保持。

## 5. 運用・品質管理
- モニタリング: 提案採用率、修正回数、作成時間短縮などをKPIで監視。
- テスト: サンプルテンプレで定期的にRAGの精度評価とABテストを実施。
- 更新運用: ソースごとにインデックス更新の頻度と責任者を定義。

## 6. 導入ステップ（MVP→拡張）
- MVP: `.docx`入出力、Markdown編集、ローカルテンプレのRAG取り込み、基本校閲ルール。
- 中期: クラウド連携、ベクトルDB整備、MCP/API連携、差分提案の高度化。
- 長期: 専用モデル／継続学習、ガバナンス自動化、組織横断展開。

## 用語定義
- RAG: retrieval-augmented generation。外部知識を検索してコンテキスト注入する手法。
- チャンク: 埋め込み対象に分割されたテキスト単位。
- source_id: 出典識別子（ファイルID/URL）。

各ソース共通の運用注意点：出典（ファイルID/URL/作成日/著者）を必ずチャンクに紐付け、更新ポリシー（即時/定期/手動）を定義、機密ソースは埋め込み前にPII検出・マスキングを行い、アクセス権限を厳格に管理します。
